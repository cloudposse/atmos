Atmos is not just another CLI, task runner, Terraform wrapper, or CI helper. It is a declarative runtime for operational orchestration across tools. That is powerful, but it creates a different problem: when a tool defines a new category, how do you show people what it is?

The answer is proof. These new demos show Atmos doing the work, and the demos themselves are generated by Atmos.

<!--truncate-->

## The Problem

Most developers already have names for the pieces around their operations stack.

They have a Terraform or OpenTofu wrapper. They have Kubernetes and Helm scripts. They have task runners, Makefiles, pre-commit hooks, CI YAML, vendoring steps, validation scripts, emulator setup, release commands, and runbooks that only work if you remember which directory to run them from.

Individually, those tools are familiar. Together, they become a pile of glue.

That is why Atmos can be hard to explain. Not because it can run Terraform, OpenTofu, Kubernetes, Helm, and ordinary development commands. Lots of tools can execute commands. Atmos is different because it treats operational automation as one cohesive system.

Workflows, custom commands, custom steps, hooks, components, auth, toolchains, vendoring, validation, local emulators, and CI behavior are not separate islands. They stack on top of the same runtime model. Once one capability exists, the next capability can compose with it instead of being bolted on beside it.

That is the category: declarative operational orchestration across tools.

## Why That Matters

The hard part of operational automation is rarely the single command. It is everything around the command:

- Preparing the right working directory.
- Creating repeatable fixtures.
- Selecting the right stack and component.
- Passing environment safely.
- Running commands locally and in CI without rewriting them.
- Capturing output humans can read.
- Cleaning up after failures.
- Validating that the automation still does what it claims.

Most teams solve those problems by assembling dozens of mature tools and a lot of less mature glue. Atmos goes after the glue directly.

If you like declarative workflows, this is the interesting part. Atmos lets the workflow describe the operation, not just the shell commands. That distinction is what lets the same model support infrastructure orchestration, routine development tasks, local demos, hooks, validation, toolchain setup, and CI behavior without each one becoming a new bespoke framework.

The power is compositional. Workflows can call Atmos commands. Custom commands can use structured steps. Hooks can participate in the same execution model. Toolchains, auth, components, and runtime behavior are available to the automation instead of being reimplemented inside scripts.

That is why the demos matter. They are not just screenshots of features. They are evidence that the runtime is broad enough to orchestrate real work.

## The Proof

The new website demos are generated by an Atmos workflow.

Not hand-recorded terminal sessions. Not one-off local recordings. The workflow builds Atmos, prepares isolated fixtures, records the casts, writes them into the website, and validates the output so committed demos stay reproducible.

You can read the workflow here:

[github.com/cloudposse/atmos/blob/main/demo/casts/atmos.yaml](https://github.com/cloudposse/atmos/blob/main/demo/casts/atmos.yaml)

That workflow currently drives demos for Terraform plan, deploy, and output; vendoring; stack listing; component and stack description; and an AWS emulator lifecycle. The point is not that every one of these is surprising in isolation. The point is that they are all expressed through one declarative operational runtime.

Here are a few representative examples.

## Vendoring

Vendoring is another place where teams often accumulate scripts. Atmos makes source acquisition part of the declared project workflow instead of a tribal-knowledge step before the real work starts.

## Stack Introspection

Operations depend on knowing what exists. Atmos can describe stacks and components with the same context it uses to run them.

## Local Emulators

The AWS emulator demo shows Atmos coordinating setup, runtime state, Terraform execution, output inspection, and teardown.

<CastPlayer src="/casts/examples/emulator-aws/lifecycle.cast" title="atmos emulator aws lifecycle" chrome controls scrubber />

This is where the category starts to become visible. A local emulator lifecycle is not just a cloud command. It touches containers, readiness, Terraform, stack context, terminal output, and cleanup. In many projects, that becomes a shell script wrapped by a Make target wrapped by CI YAML. In Atmos, it can be part of the same declarative model as everything else.

## Proof Beats Promises

When you pave a new road, education is part of the work. People need familiar entry points, but they also need to see the shape of the new category.

That is what these demos are for. They show Atmos orchestrating Terraform and OpenTofu, vendoring, stack introspection, local emulators, and routine development flows. More importantly, they show Atmos using Atmos to produce and verify the demos themselves.

For developers who like declarative workflows, this is the pitch: stop scattering operational behavior across scripts, CI-only config, local setup notes, and one-off helpers. Put it in one runtime that understands your stacks, components, tools, hooks, commands, auth, and workflows as parts of the same system.

That is what makes Atmos different. Not one feature. The composition.
