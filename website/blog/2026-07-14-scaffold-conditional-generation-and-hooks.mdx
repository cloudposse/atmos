Code generators usually force a choice: keep the template dead simple so every generated project
looks identical, or maintain a pile of near-duplicate templates for every combination of options
you support. Tools like Cookiecutter popularized conditional file rendering to get around this;
Copier went further, letting you re-apply a template later and merge in upstream changes without
clobbering the customizations you've already made. Atmos scaffold templates now support both
halves of that story — plus a way to run something automatically once generation finishes.

<!--truncate-->

## The Problem

A `scaffold.yaml` template's fields and files were static. Every field got prompted for, every
file got generated, regardless of what the user actually needed. Want an optional monitoring
stack, or an extra environment, or a vendoring manifest? Either bake it in for everyone, strip it
out by hand afterward, or maintain a second template. And once generation finished, that was it —
no way to declare "after this runs, format the generated files" or "before this runs, check
something first."

## The Fix

Three additions to the `scaffold.yaml` schema, all reusing pieces Atmos already had:

- **Conditional prompts** — a field can declare `when:`, so it's only asked if an earlier answer
  makes it relevant. Skip the "alert email" prompt entirely unless "enable monitoring" was
  answered yes.
- **Conditional file generation** — a file (or whole directory) can declare `when:` too, so it's
  only written when the condition holds. Everything not listed still generates unconditionally, as
  before — this only applies to files you explicitly gate.
- **Generation hooks** — a template can declare `hooks:` that run before or after generation,
  gated by the same `when:` condition engine Atmos workflows and CI hooks already use. `--skip-hooks`
  opts out per invocation, either entirely or by name.

Both `when:` forms accept the same condition language used everywhere else in Atmos (a bare
predicate, or a CEL expression), with one addition: an `answers` variable exposing everything
prompted for so far.

Both `atmos init` and `atmos scaffold generate` are doors into the same engine, so all three of
these work identically from either command.

## How to Use It

A conditional prompt and a conditional file, chained together:

```yaml
spec:
  fields:
    - name: enable_vendoring
      label: Enable component vendoring
      type: confirm
      default: true

    - name: vendor_version
      label: Vendor component version
      type: input
      default: "1.536.0"
      when: "answers.enable_vendoring == true"

  files:
    - path: vendor.yaml
      when: "answers.enable_vendoring == true"
```

A generation hook that formats output afterward:

```yaml
hooks:
  format:
    events:
      - after.scaffold.generate
    kind: step
    type: shell
    with:
      command: "terraform fmt"
```

Try the conditional-prompt/conditional-file combination above for real, straight from GitHub —
no local clone needed:

```shell
# vendor_version is asked, and vendor.yaml is generated with a pinned source
atmos scaffold generate github.com/cloudposse/atmos.git//examples/scaffolding ./my-project \
  --set enable_vendoring=true --set vendor_version=1.536.0

# vendor_version is never asked, vendor.yaml is never generated
atmos scaffold generate github.com/cloudposse/atmos.git//examples/scaffolding ./my-project \
  --set enable_vendoring=false
```

### Works with GitHub template repositories, and goes further

Any plain Git repository works as a scaffold source — including one marked as a GitHub "template
repository," which most teams already use to bootstrap new projects with GitHub's "Use this
template" button. Point `atmos scaffold generate` (or `atmos init`) at it the same way, optionally
pinned to a branch, tag, or commit with `--ref`.

Where this goes further than a one-time template copy: `atmos scaffold generate --update` re-runs
the same source against an existing project and 3-way merges the result — upstream template
changes land without silently overwriting the customizations you've already made, with
`--merge-strategy` (`manual`/`ours`/`theirs`) controlling how real conflicts get resolved. A GitHub
template repository's "Use this template" button gives you a one-time copy with no relationship
back to the source afterward; `--update` keeps that relationship alive.
