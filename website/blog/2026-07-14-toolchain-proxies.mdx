Some releases package more than one command behind a single binary. uutils is a good example: the Aqua registry installs `coreutils`, but the command you want to type is `ls`. Atmos toolchain proxies close that gap without asking every developer to maintain shell aliases or copied shims.

<!--truncate-->

## A familiar command backed by a pinned tool

Configure an alias, a pinned version, and a proxy:

```yaml title="atmos.yaml"
toolchain:
  aliases:
    coreutils: uutils/coreutils
  proxies:
    ls:
      tool: coreutils
      args: [ls]
```

```text title=".tool-versions"
coreutils 0.9.0
```

Then activate the toolchain in your shell and use `ls` normally:

```shell
eval "$(atmos toolchain env)"
ls stacks/
```

The first invocation installs the pinned `uutils/coreutils` release if it is not already present. Atmos forwards the command as `coreutils ls stacks/`; later calls reuse the installed binary.

## How it works

The design is inspired by Aqua's proxy mechanism. Atmos creates a command-named link in its toolchain proxy directory. When that link starts Atmos as `ls`, Atmos recognizes its own `$0`, resolves the configured proxy, and runs the underlying tool with the original arguments.

This also means the feature is useful beyond an interactive shell. Built-in Atmos command runners, workflows, hooks, components, and custom commands inherit the proxy environment automatically. A command can use `ls`, or another configured proxy, while still using the version-controlled toolchain definition.

The override is deliberately scoped. Atmos never changes your system `ls`; the proxy is available only to Atmos child processes or a shell where you opted in with `atmos toolchain env`.
