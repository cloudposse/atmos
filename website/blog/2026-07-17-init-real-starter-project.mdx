Platform teams need every new service, environment, and tenant to begin on a paved road. GitHub
template repositories are excellent at distributing a starting tree, but a copy alone cannot ask
which environments a team needs, enforce the platform's naming rules, tailor the generated files,
or prove the result works. The first developer still has to connect the pieces and discover the
gaps.

`atmos init` turns a starter template into a guided, executable project bootstrap. It captures the
choices that matter, generates only the right project shape, and gives the team a working path from
day one.

<!--truncate-->

## The Platform Problem

At organization scale, "start from this repository" is not a platform experience. A static copy
cannot prevent bad inputs from slipping through automation, distinguish a small development setup
from a multi-environment project, or establish confidence that the generated repository is ready
for a real workflow. That creates template sprawl, inconsistent first commits, and a support queue
for the platform team.

## What `init` Delivers Now

The cloud-agnostic `basic` template now starts with a real `greeting` Terraform component: one
local `local_file` resource, with no cloud account, credentials, or emulator required. It is small
on purpose. The first outcome is proof that the project can run, not an empty directory asking the
developer to build that proof themselves.

The template asks which environments to create and generates only those stacks. Its field rules
are enforced consistently whether values come from an interactive answer, a default, a saved
project value, or `--set`, so the same template works for a developer at a terminal and for
repeatable automation.

For platform teams, that means a starter project can carry both the organization’s conventions and
an executable acceptance path. Developers get a useful first commit; the platform gets a governed,
repeatable entry point instead of another one-off repository copy.

`init` and `scaffold generate` use the same generation engine. The conditional fields, files, and
hooks described in [the scaffold announcement](/changelog/scaffold-conditional-generation-and-hooks)
therefore apply to a whole-project bootstrap as well as to a catalog template. This is what turns a
GitHub template repository from a useful starting artifact into part of a platform workflow.
