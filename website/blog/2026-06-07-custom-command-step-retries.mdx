Custom command steps now support a `retry` block, so a step that fails with a transient error (a flaky network call, a registry returning an intermittent `5xx`) is automatically re-run with configurable backoff instead of failing the whole command.

<!--truncate-->

## What Changed

Custom commands defined in `atmos.yaml` already let you script multi-step workflows. Now each step can declare its own `retry` policy — the same `RetryConfig` already used by [workflow steps](/workflows#retry-configuration). Write the step in its structured form and add a `retry` block:

```yaml
commands:
  - name: helm
    description: Execute 'helm' helper commands
    commands:
      - name: plugins
        description: Manage Helm plugins required by Atmos
        commands:
          - name: install
            description: Install the helm-diff plugin
            steps:
              - name: install-helm-diff
                command: helm plugin install https://github.com/databus23/helm-diff
                retry:
                  max_attempts: 3
                  backoff_strategy: exponential
                  initial_delay: 5s
                  max_delay: 30s
```

A step is retried on **any** non-zero exit, governed by `max_attempts`, the backoff settings, and `max_elapsed_time`. Steps without a `retry` block run exactly once, with no change to their behavior or error output — so this is fully backward compatible.

## Why This Matters

Before this change, the only way to make a flaky step resilient was to hand-roll a retry loop inside the command's shell. That logic is easy to get wrong and clutters every call site. Custom command steps and workflow steps now share one retry engine, with one consistent set of knobs (`max_attempts`, `backoff_strategy`, `initial_delay`, `max_delay`, `multiplier`, `random_jitter`, `max_elapsed_time`).

We dogfood it ourselves: Atmos's own CI installs the `helm-diff` plugin via an `atmos helm plugins install` custom command, which now rides out the occasional transient registry error instead of failing the build.

## How to Use It

Add a `retry` block to any `shell` or `atmos` step you want to make resilient. See the [custom commands reference](/cli/configuration/commands#step-retries) for the full list of options.
