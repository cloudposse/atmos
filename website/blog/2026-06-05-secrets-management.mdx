Atmos now has first-class **secrets management** — a GitOps-friendly, multi-cloud workflow for declaring,
provisioning, and resolving secrets, with automatic masking everywhere.

<!--truncate-->

## What Changed

You can now declare secrets in your stack config and manage their values with a Vercel-like CLI:

```shell
atmos secret set DATADOG_API_KEY --stack=prod --component=api
atmos secret list --stack=prod --component=api
atmos secret validate --stack=prod --component=api
```

Secrets are **declared** (committed to git) and their **values** live in a cloud secret backend or a
SOPS-encrypted file. They are resolved at runtime with the new `!secret` YAML function:

```yaml
components:
  terraform:
    api:
      secrets:
        vars:
          DATADOG_API_KEY:
            store: app-secrets       # a `secret: true` store
            required: true
      vars:
        datadog_api_key: !secret DATADOG_API_KEY
```

**Backends.** Track 1 reuses the existing store registry — a store becomes a secret backend by setting
`secret: true`. Supported backends include AWS SSM Parameter Store (SecureString), the new **AWS Secrets
Manager** and **HashiCorp Vault** store types, Azure Key Vault, and GCP Secret Manager. Track 2 adds a native
**SOPS** provider (`age`, `aws-kms`, `gcp-kms`, `gpg`) for git-committed encrypted files.

**CLI.** The full CRUD surface ships: `init`, `set` (alias `add`), `get`, `delete` (alias `rm`), `list`,
`pull`, `push`, `import`, and `validate`.

## Why This Matters

Before this, there was no unified way to manage human-provisioned secrets in Atmos. Stores were designed for
machine-written Terraform outputs, and the historical workaround (Chamber) was AWS-only. Worse, secrets tended
to **sprawl** — inheriting dev and staging secrets into prod pipelines.

Atmos secrets fix this with explicit, declarative registration: a secret must be declared in a component's
`secrets.vars` before it can be set, read, or resolved. `!store` against a `secret: true` store is an error
("use `!secret`"), so there is never any ambiguity about whether a value is sensitive.

## Inspect Without Credentials

The headline feature is the masking integration. Inspection commands — `atmos describe` and the `atmos list`
family — resolve `!secret` to `<MASKED>` **without contacting the backend** when masking is enabled (the
default). That means you can diff or review a stack's shape in CI or on a laptop with **no cloud access**:

```shell
# No credentials needed — the secret renders as <MASKED>
atmos describe component api --stack=prod

# Reveal the real value (requires access to the secret backend)
atmos describe component api --stack=prod --mask=false
```

Value-producing commands (`atmos secret get`, `terraform plan`/`apply`) always retrieve the real value, and
the global `--mask` flag only controls whether it is redacted in display output. Sensitive Terraform outputs
(`sensitive = true`) are now also automatically registered with the masker as they flow through
`!terraform.output`, `atmos.Component()`, and `atmos describe`.

## How to Use It

1. Configure a `secret: true` store in `atmos.yaml`.
2. Declare your secrets under a component's `secrets.vars`.
3. Provision values with `atmos secret set` (or `init` to be prompted for missing ones).
4. Reference them with `!secret NAME` in your component vars.
5. Verify with `atmos secret list` and gate CI with `atmos secret validate`.

See the [secrets configuration guide](/cli/configuration/secrets), the
[`atmos secret` command reference](/cli/commands/secret/usage), and the
[`!secret` YAML function](/functions/yaml/secret) to get started.
