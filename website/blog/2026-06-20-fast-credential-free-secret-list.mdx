`atmos secret list` now requires **no authenticated identity and never decrypts** — it only reports whether each secret is *set*. On a real-world stack of 72 components, listing dropped from ~21–34 seconds (and often hanging while it authenticated every component) to effectively instant.

<!--truncate-->

## What Changed

Listing secrets is an introspection operation: it shows you which secrets are declared and whether they're initialized — it never needs a plaintext value. But the old implementation authenticated every component's identity during stack processing and, to populate the `STATUS` column, fetched and **decrypted** every secret just to check that it existed.

This release makes listing credential-free by default:

- **No per-component authentication.** Enumerating declarations no longer runs an auth cycle per component.
- **SOPS status without decryption.** A SOPS-backed secret's `initialized` status is read from the file's cleartext key names — no age key, no decryption.
- **Store existence checks never decrypt.** Every backend's existence check now uses a metadata API (AWS SSM `GetParameter` with decryption off, Secrets Manager `DescribeSecret`, GCP `GetSecretVersion`, Azure Key Vault properties, Vault KV metadata, 1Password without revealing the value) instead of fetching the secret. Listing no longer needs `kms:Decrypt`-style permissions.

## Why This Matters

The `STATUS` column on a remote store still needs a credential just to ask "does this exist?" — so by default those secrets now show `unknown` rather than forcing you to authenticate. Local backends like SOPS always show an accurate status for free. The result: a `secret list` that's fast, safe to run anywhere (CI, a fresh checkout, no cloud login), and never decrypts data you didn't ask for.

## How to Use It

```shell
# Credential-free by default — SOPS shows real status, remote stores show "unknown"
atmos secret list --stack=prod

# Want an authoritative status for a remote-backed secret? Opt in with --verify
atmos secret list --stack=prod --component=api --verify
```

`--verify` contacts remote backends using a read/describe identity (never a decrypt identity) for a definitive `initialized`/`missing` answer. It applies to a fully-scoped (`--stack` + `--component`) target.
