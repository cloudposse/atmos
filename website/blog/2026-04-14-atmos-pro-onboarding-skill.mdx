Setting up Atmos Pro in an existing repo means assembling GitHub OIDC trust, two auth
profiles, four GitHub workflows, an IAM role catalog, a `settings.pro` mixin, and reciprocal
tfstate-backend trust — across every target account. Until now, that was a hand-written,
multi-day project. The new `atmos-pro` agent skill turns it into a handful of prompts.

<!--truncate-->

## Why This Matters

Every repo starts from a different place: some have Atmos Auth, some don't; some use
Spacelift, some use Geodesic; GitHub Actions may or may not be enabled; federated IAM
may come from Identity Center or an external IdP. Each variant changes what needs to
be generated.

The `atmos-pro` skill encodes the full contract — detection, generation, validation,
PR creation — and produces a deploy-ready PR regardless of the starting shape.

## One Skill, Two Invocation Paths

### Path A — Atmos-dispatched

```bash
atmos ai ask "setup atmos pro" --skill atmos-pro
```

Atmos loads the skill from its embedded bundle and runs the playbook through the AI provider
configured in `atmos.yaml`. No prior `atmos ai skill install` required — the skill ships with
the binary. This path requires [Atmos AI](/ai) to be configured.

### Path B — Claude Code direct

Install the marketplace plugin once, then ask Claude Code to run the skill:

```text
/plugin marketplace add cloudposse/atmos
/plugin install atmos@cloudposse
```

Then in the chat: "Set up Atmos Pro for this repo using the atmos-pro skill."

Claude Code loads `SKILL.md`, pulls reference files on demand, and executes using its native
Read/Write/Edit/Bash tools. Output is byte-identical to Path A — both paths share the same
templates and golden-snapshot contract.

## What the Skill Does

The flow is strict:

1. **Creates an isolated worktree** so your main checkout is never touched.
2. **Detects the repo shape** — stack hierarchy, Atmos Auth presence, Spacelift usage,
Geodesic environment, `github-oidc-provider` deployment, tfstate backend trust.
3. **Confirms the plan** — prints which org, which accounts, which files will change, and
which starting-condition variants apply. You approve before anything is written.
4. **Generates artifacts** from templates, filled in from the detected values:
- `stacks/mixins/atmos-pro.yaml` — single import point, disables Spacelift, declares
`settings.pro`
- `stacks/catalog/aws/iam-role/{defaults,gha-tf}.yaml` — abstract base plus two
OIDC-trusted concrete roles
- `components/terraform/aws/iam-role/component.yaml` — vendor manifest
- `profiles/github-plan/atmos.yaml`, `profiles/github-apply/atmos.yaml` —
per-account identities (root account pinned to the plan role for safety)
- `.github/workflows/` — four workflows (affected stacks, list instances, plan, apply)
plus an optional OIDC smoke test
- Additive edit to `stacks/catalog/tfstate-backend/defaults.yaml` — wildcard
`allowed_principal_arns`
- `docs/atmos-pro.md` — generated README with your rollout procedure
5. **Self-validates** — runs `atmos validate stacks` and `atmos describe component` on the
generated config. Stops on failure. No PR opened with a broken worktree.
6. **Opens a PR** with a rollout checklist body (deploy IAM roles per account, deploy the
tfstate trust update, run the OIDC test workflow, merge).

## Starting-Condition Awareness

The skill branches on every variant we've seen onboarding fail on:

| Variant                                      | What the skill does                                                           |
|----------------------------------------------|-------------------------------------------------------------------------------|
| GitHub Actions disabled org-wide             | Stops, prints the exact settings URL to enable it                             |
| No Atmos Auth in the repo                    | Generates standalone profiles, does not retrofit `auth:` into `atmos.yaml`    |
| Atmos Auth already configured                | Adds the `github-oidc` provider via a patch file, preserves existing config   |
| Federated IAM via Okta (not Identity Center) | Generates CI-side profiles only, documents that local dev continues with Okta |
| Geodesic-based dev environment               | Adds a Geodesic section to the generated docs with `GITHUB_TOKEN` passthrough |
| Spacelift currently enabled                  | Disables Spacelift in the mixin, flags the migration in the PR body           |
| `github-oidc-provider` not deployed          | Adds a step-0 to the rollout checklist                                        |
| Custom tfstate `allowed_principal_arns`      | Merges additively — never overwrites your existing principals                 |

## Safety Rails You Can Trust

These are non-negotiable defaults baked into the templates:

- **Default-restrictive branch scoping** — `trusted_github_repos` defaults to `main`-only.
The planner role explicitly opts out (needs PR branches); the apply role never does.
- **Root account always-plan** — the apply profile points the root account's identity at
the plan role's ARN, with a comment explaining why. Automation never applies to root.
- **Reciprocal trust ships together** — the IAM role changes and the tfstate-backend edit
appear in the same PR. The skill refuses to generate one without the other.
- **No secrets in manifests** — `ATMOS_PRO_WORKSPACE_ID` is a repo **variable**, not a
secret.

## Built-In, Not Installable

The `atmos-pro` skill ships embedded in the Atmos binary alongside the other
built-in skills — no separate install step:

```bash
$ atmos ai skill list

Built-in skills:
✓ atmos-pro
  Atmos Pro onboarding: generate GitHub OIDC IAM roles, auth profiles, CI workflows,
  mixins, and open a PR that wires an Atmos repo to Atmos Pro
✓ atmos-terraform
  Terraform orchestration: plan/apply/deploy, workspace management, backend config, ...
```

You can still override a built-in with a marketplace install — marketplace versions take
precedence for the same skill name.

## Under the Hood

The skill is backed by testable Go packages so the output is deterministic regardless of
which AI drives it:

- Filesystem-based pre-flight probes (Atmos Auth, Spacelift,
Geodesic). Pure Go, `fs.FS`-based, no external binaries.
- Template renderer with a typed context and
custom `<<...>>` delimiters so `{{ }}` and `${{ }}` in the output pass through untouched.
- Exposes every built-in skill to the registry.
- **Golden-snapshot fixtures** guarantee rendering is byte-stable across refactors.
- **Cross-path parity tests** prove the on-disk and embedded templates are byte-identical.

## Real-World Run — 4 Prompts From Zero to Deploy-Ready PR Using Claude Code

We ran the skill end-to-end against a multi-org, Geodesic-hosted Atmos refarch.

**Install the skill (one-time):**

```text
/plugin marketplace add cloudposse/atmos
/plugin install atmos@cloudposse
/reload-plugins
```

**Run 1 — detect:**

> Using `atmos:atmos-pro`, run the detection phase. Report stack hierarchy,
> Atmos Auth, Spacelift, github-oidc-provider, Geodesic, GitHub Actions.

→ Full inventory returned as a structured table.

**Run 2 — plan:**

> Using `atmos:atmos-pro`, show the plan summary. Do not write.

→ Target scope, accounts table, file-create / file-edit lists, starting-condition
variants, and open policy questions for approval.

**Run 3 — generate:**

> Proceed: resolve the account map, enable all orgs, pin prod to plan-only,
> create the worktree, generate, validate.

→ Worktree at `.worktrees/atmos-pro-setup`, all artifacts staged and validated.
All safety rails applied.

**Run 4 — open the PR:**

> Open the PR with `gh pr create --draft`, using the repo's
> `PULL_REQUEST_TEMPLATE.md` sections.

→ Draft PR opened with body mapped to the repo's template.

### The result

A draft PR with the full artifact set, body matching the repo's template, and
every safety rail honored — ready for review and rollout.

## Get Started

1. Set your Atmos Pro workspace ID as a GitHub repo variable:

```bash
gh variable set ATMOS_PRO_WORKSPACE_ID --body "<your-workspace-id>"
gh variable set ATMOS_VERSION --body "<atmos-version>"
   ```

2. Run the skill:

```bash
atmos ai ask "setup atmos pro" --skill atmos-pro
   ```

3. Review the generated PR and follow the rollout checklist.
