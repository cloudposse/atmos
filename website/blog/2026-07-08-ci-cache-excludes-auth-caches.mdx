If you've turned on `ci.cache` to warm-start your toolchain across CI runs, it was quietly
caching more than your toolchain. The default "cache the whole Atmos cache directory" behavior
also swept up `~/.cache/atmos/aws-sso`, `azure-device-code`, and `aws-webflow` — the
directories where Atmos's own auth flows keep session tokens and refresh tokens on disk.

<!--truncate-->

## The Problem

`ci.cache` restores and saves one well-known directory (`~/.cache/atmos` by default) so the
toolchain and other regenerable data survive across CI jobs. When `ci.cache.paths` was left
unset — the common case — Atmos archived that entire directory with no exclusions. That meant
AWS SSO access/refresh tokens and OAuth client secrets, Azure device-code tokens, and Atmos's
browser-based AWS webflow refresh token could end up bundled into the same cache archive
uploaded to your CI provider's cache store, right alongside the toolchain binaries.

To be clear: this wasn't a disclosed exploit. Access to a repo's GitHub Actions cache is
already scoped to that repo, and everything above is short-lived, rotating session material
rather than a static long-lived secret. But it was a foot-gun — nothing stopped a whole-root
cache from bundling credentials, and that risk shouldn't depend on every `ci.cache.paths`
configuration remembering to avoid the right subdirectories by hand. OIDC-based auth
(GitHub/GCP/Azure) was unaffected either way — those flows mint credentials fresh in-memory
every run and never touch the cache directory.

## The Fix

Atmos now excludes `aws-sso`, `azure-device-code`, `aws-webflow`, and `auth` from the CI cache
unconditionally — regardless of what you put in `ci.cache.paths` — in both Atmos's own cache
backend and the `atmos ci cache paths` output used with the native `actions/cache` GitHub
Action. If you've already been using `ci.cache`, any previously saved cache entries may still
contain session material; consider running `atmos ci cache delete` on them, or just let them
age out naturally as your cache key rotates.

## How to Use It

No changes are needed to pick up the safer default. If you have a specific, trusted reason to
cache your own credential material anyway, opt back in explicitly:

```yaml
ci:
  cache:
    enabled: true
    allow_unsafe_auth_cache: true # opt out of the default exclusion
```

See the [CI Cache configuration reference](/cli/configuration/ci/cache) for the full list of
excluded paths and the `ATMOS_CI_CACHE_ALLOW_UNSAFE_AUTH_CACHE` environment variable.
