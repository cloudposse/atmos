<Intro>
This [Hands-on Lab](/labs) is a complete, copy-and-run project that builds a hardened
**Amazon Linux 2023** AMI with **Packer** orchestrated by **Atmos**, validates it on a
live test instance, gates promotion behind a **manual approval**, tags the approved image
`ScanStatus=approved`, and **shares it across AWS accounts** — all driven from a GitHub
Actions pipeline and a set of `atmos ami` custom commands.
</Intro>

<KeyPoints title="What you'll learn">
- How to configure and run **Packer components** with Atmos.
- How to drive a Packer build entirely from **stack configuration** (no hardcoded HCL).
- How to build a tree of **nested custom commands** (`atmos ami ...`) around your images.
- How to wire a **governed CI/CD pipeline** with OIDC auth and a manual approval gate.
- How to enforce **"launch only approved AMIs"** with a tag-based IAM/SCP policy.
</KeyPoints>

## Architecture

```text
        ┌──────────────────────────────────────────────────────────────┐
        │ GitHub Actions Pipeline (.github/workflows/ami.yml)           │
        ├──────────────────────────────────────────────────────────────┤
        │ build (Packer via Atmos) → launch test instance               │
        │   → health check → [optional] scan                            │
        │   → ⏸ manual approval gate (GitHub Environment)               │
        │   → tag ScanStatus=approved → share AMI → cleanup             │
        └───────────────┬──────────────────────────────────────────────┘
                        │ atmos packer build / atmos ami …
        ┌───────────────▼──────────────────────────────────────────────┐
        │ Atmos: Packer component + stack + `atmos ami` custom commands │
        └───────────────┬──────────────────────────────────────────────┘
                        │ packer build
        ┌───────────────▼──────────────────────────────────────────────┐
        │ Packer (amazon-ebs): patch → harden → [scan] → install → finalize │
        └──────────────────────────────────────────────────────────────┘
```

## 1. Configure Atmos and the custom commands

The `atmos.yaml` does two jobs: it registers the Packer component type, and it defines a
nested `atmos ami <subcommand>` command tree. Each subcommand takes the Packer `component`
and `--stack`, exposes resolved stack values (region, KMS key) to the script via
`component_config`, and delegates the work to a small, reviewable script.

<EmbedFile filePath="labs/aws-ami-packer-github-actions/atmos.yaml" language="yaml" />

<Note>
The `atmos ami get-ami-id` command reads the AMI ID from the Packer build **manifest**
using `atmos packer output ... -q '<yq>'`, selecting the build whose `packer_run_uuid`
matches the manifest's `last_run_uuid`. That makes it robust even when the manifest
contains several historical builds.
</Note>

## 2. Parameterize the build with a stack

Everything the build needs is configuration in the stack — source AMI, networking,
encryption, sharing targets, tags, and the ordered list of provisioner scripts. The
`source_ami_name` resolves from the `SOURCE_AMI_NAME` environment variable at build time
(via Go templating), so CI can pin an exact base image while local builds fall back to the
latest. Every account-specific value is a clearly marked **PLACEHOLDER**.

<EmbedFile filePath="labs/aws-ami-packer-github-actions/stacks/al2023.yaml" language="yaml" />

## 3. The Packer template

The `main.pkr.hcl` is intentionally generic: it declares variables for every input and
reads them from the stack. It discovers the base AMI with a `data.amazon-ami` lookup
(no brittle AMI IDs), runs the stack-provided scripts as ordered shell provisioners, and
emits a `manifest.json` the custom commands read.

<EmbedFile filePath="labs/aws-ami-packer-github-actions/components/packer/al2023/main.pkr.hcl" language="hcl" />

## 4. The governed pipeline

The GitHub Actions workflow chains the stages and inserts a **manual approval gate** using
a GitHub Environment (`ami-approval`). The AMI is only tagged `approved` and shared *after*
a human approves it. Authentication uses **OIDC** — no long-lived AWS secrets.

<EmbedFile filePath="labs/aws-ami-packer-github-actions/.github/workflows/ami.yml" language="yaml" />

## 5. Enforce "launch only approved AMIs"

Tagging approved images is only half the story — you also want to *enforce* that only
approved images can be launched. This reference SCP denies `ec2:RunInstances` unless the
AMI carries `ScanStatus=approved` (and denies it outright when the tag is missing, so it
fails closed).

## Run it

```bash
# Copy the lab into a new repo
cp -r labs/aws-ami-packer-github-actions/ my-ami-pipeline/
cd my-ami-pipeline/

# Build locally (after editing stacks/al2023.yaml for your environment)
atmos packer init  al2023 -s al2023
atmos packer build al2023 -s al2023

# Operate the result
atmos ami get-ami-id      al2023 -s al2023
atmos ami launch-instance al2023 -s al2023 --type t3.small
atmos ami tag             al2023 -s al2023 --key ScanStatus --value approved
atmos ami share           al2023 -s al2023 --accounts 123456789012,123456789013
```
