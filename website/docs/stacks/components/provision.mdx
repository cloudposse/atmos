<Intro>
The `provision` section controls automatic, just-in-time creation of supporting infrastructure (Terraform state backends) and execution environments (isolated workdirs) before component commands run. Use it to remove manual bootstrap steps and to enable safe concurrent execution.
</Intro>

<Experimental />

## Use Cases

- **Backend Bootstrap:** Automatically create the Terraform state backend (S3 bucket, lockfile) the first time a component runs, eliminating the chicken-and-egg bootstrap problem.
- **Isolated Execution:** Run each component instance in its own working directory so concurrent operations on different components do not collide on `.terraform/`, lockfiles, or generated varfiles.
- **Just-in-Time Provisioning:** Fetch remote component sources on demand into a per-instance workdir on fresh CI runners.

## Configuration Scope

The `provision` section can appear at three levels and is deep-merged across them:

- **Component level** тАФ `components.<type>.<name>.provision`
- **Toolchain defaults** тАФ `terraform.provision`, `helmfile.provision`, `packer.provision`, `ansible.provision`
- **Global defaults** тАФ `settings.provision` in `atmos.yaml` (workdir only, today)

Lower levels override higher levels, so a component can opt out of a toolchain default by setting `enabled: false`.

## Schema

The `provision` block has two sub-keys:

<dl>
  <dt>`provision.backend.enabled`</dt>
  <dd>
    Automatically provision the Terraform state backend before the first run.
    Currently supports S3 backends with optional native S3 locking (Terraform 1.10+).
    See [Backend Provisioning](/components/terraform/backend-provisioning) for the full conceptual treatment.

    **Applies to:** Terraform components only.
    **Default:** `false`.
  </dd>

  <dt>`provision.workdir.enabled`</dt>
  <dd>
    Create an isolated working directory for each component instance at
    `.workdir/<componentType>/<stack>-<componentName>/`. The component source
    is staged into this directory and all toolchain commands execute there,
    enabling concurrent execution and just-in-time source provisioning.

    **Applies to:** Terraform, Helmfile, Packer, and Ansible components.
    **Default:** `false`.
  </dd>
</dl>

## Provisioning the Backend

Enable automatic backend creation alongside the existing `backend:` configuration. The `backend:` block describes *where* state lives; `provision.backend.enabled` tells Atmos to *create that location* if it does not exist.

<File title="stacks/catalog/vpc.yaml">
```yaml
components:
  terraform:
    vpc:
      backend_type: s3
      backend:
        bucket: acme-ue1-dev-tfstate
        key: vpc/terraform.tfstate
        region: us-east-1
        use_lockfile: true  # Native S3 locking (Terraform 1.10+)

      provision:
        backend:
          enabled: true  # Auto-create the bucket if missing
```
</File>

For a deep dive on what is provisioned, secure defaults, and how to import an existing backend, see [Backend Provisioning](/components/terraform/backend-provisioning).

## Provisioning the Workdir

When workdir provisioning is enabled, Atmos creates a per-component-instance directory under `.workdir/` and runs all toolchain commands there. Each component instance gets its own `.terraform/`, varfiles, and state cache, so multiple components can run in parallel.

`provision.workdir.enabled` works for every supported toolchain:

<File title="stacks/catalog/_defaults.yaml">
```yaml
components:
  terraform:
    vpc:
      provision:
        workdir:
          enabled: true
      vars:
        cidr_block: "10.0.0.0/16"

  helmfile:
    nginx:
      provision:
        workdir:
          enabled: true
      vars:
        replicas: 3
```
</File>

The directory layout looks like this:

```
.workdir/
тФЬтФАтФА terraform/
тФВ   тФФтФАтФА prod-ue1-vpc/        # <stack>-<componentName>
тФВ       тФЬтФАтФА .terraform/
тФВ       тФЬтФАтФА .atmos/metadata.json
тФВ       тФФтФАтФА ... (component source)
тФФтФАтФА helmfile/
    тФФтФАтФА prod-ue1-nginx/
        тФФтФАтФА ...
```

Manage workdirs with the [`atmos terraform workdir`](/cli/commands/terraform/workdir) commands (`list`, `show`, `clean`, `describe`).

## Setting Defaults at the Toolchain Level

Apply provisioning to every component of a toolchain by setting it under the toolchain key:

<File title="stacks/orgs/acme/plat/dev/_defaults.yaml">
```yaml
terraform:
  provision:
    backend:
      enabled: true   # Every Terraform component in this stack gets backend provisioning
    workdir:
      enabled: true   # ...and isolated workdirs

helmfile:
  provision:
    workdir:
      enabled: true   # Every Helmfile component runs in its own workdir
```
</File>

Toolchain-level defaults flow down to every component in the stack and can be overridden per component.

## Component-Level Overrides

Opt a single component out of a toolchain default by setting `enabled: false`:

<File title="stacks/orgs/acme/plat/dev/us-east-1.yaml">
```yaml
components:
  terraform:
    legacy-vpc:
      provision:
        workdir:
          enabled: false   # Run in the original component directory
      vars:
        # ...
```
</File>

## Global Defaults in `atmos.yaml`

Set workdir defaults globally for every stack and component via the `settings.provision` block:

<File title="atmos.yaml">
```yaml
settings:
  provision:
    workdir:
      enabled: true       # Default for every component, every stack
      ttl: "7d"           # Auto-clean workdirs not accessed for 7 days
```
</File>

<dl>
  <dt>`settings.provision.workdir.enabled`</dt>
  <dd>Global default for `provision.workdir.enabled`. Stack-level and component-level settings still override this.</dd>

  <dt>`settings.provision.workdir.ttl`</dt>
  <dd>Time-to-live for workdirs (e.g., `"7d"`, `"24h"`, `"weekly"`). Workdirs not accessed within this duration become candidates for cleanup by [`atmos terraform workdir clean`](/cli/commands/terraform/workdir).</dd>
</dl>

There is no global `settings.provision.backend` today; backend provisioning is opted in per component or per toolchain.
