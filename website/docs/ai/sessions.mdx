<Intro>
Atmos AI supports persistent conversation sessions that maintain message history across CLI invocations.
You can have long-running conversations with the AI assistant and pick up exactly where you left off,
even days or weeks later.
</Intro>

<Experimental />

:::tip Configuration Reference
For a concise reference of all session settings, see [CLI Configuration > AI > Sessions](/cli/configuration/ai/sessions).
:::

## Overview

Sessions give you persistent history stored locally in SQLite, named conversations organized by topic,
full conversation memory so the AI remembers what you discussed, and configurable retention policies
for automatic cleanup. This works across all seven supported AI providers (Anthropic, OpenAI, Google,
xAI, Ollama, AWS Bedrock, and Azure OpenAI).

## Conversation Memory

When you send a message, the AI receives the complete conversation history for the session. This
enables multi-turn conversations where you can ask follow-up questions like "yes", "tell me more",
or "what about staging?" and the AI understands the context.

### Example: Multi-Turn Conversation

<Terminal title="atmos ai chat --session vpc-design">
```bash
You: What are Atmos stacks?

AI: Atmos stacks are configuration files that define your infrastructure.
    They use inheritance and composition to share configuration across environments.

You: can you show me an example for production?

AI: Sure! Based on the stack structure I explained, here's a production example:
    # AI references the earlier explanation
    [Shows YAML example with production values]

# Day 2: Resume the session
atmos ai chat --session vpc-design

You: let's continue with the staging variant

AI: Welcome back! Based on the production stack we discussed yesterday,
    here's how staging differs:
    # AI recalls full context from previous day
    [Shows staging example highlighting differences from production]
```
</Terminal>

:::info Privacy Note
When resuming a session, the complete message history is sent to the AI provider. If your conversations
contain sensitive information, review the [Privacy and Security](#privacy-and-security) section below.
:::

## Configuration

Enable sessions in your `atmos.yaml`:

<File title="atmos.yaml">
```yaml
settings:
  ai:
    enabled: true
    default_provider: "anthropic"
    providers:
      anthropic:
        model: "claude-sonnet-4-6"
        api_key: !env "ANTHROPIC_API_KEY"

    sessions:
      enabled: true                    # Enable persistent sessions
      storage: "sqlite"                # Storage backend (default: sqlite)
      path: ".atmos/sessions"          # Storage directory (default: .atmos/sessions)
      max_sessions: 10                 # Max sessions per project (default: 10)
      auto_save: true                  # Auto-save messages (default: true)
      retention_days: 30               # Retention period in days (default: 30)
```
</File>

### Configuration Options

<dl>
  <dt>`sessions.enabled`</dt>
  <dd>Enable or disable persistent sessions (default: `false`)</dd>

  <dt>`sessions.storage`</dt>
  <dd>Storage backend for sessions. Currently only `sqlite` is supported (default: `sqlite`)</dd>

  <dt>`sessions.path`</dt>
  <dd>Directory path where the session database is stored (default: `.atmos/sessions`)</dd>

  <dt>`sessions.max_sessions`</dt>
  <dd>Maximum number of sessions to keep per project. When this limit is reached, the oldest sessions are automatically cleaned up (default: `10`)</dd>

  <dt>`sessions.auto_save`</dt>
  <dd>Automatically save messages after each exchange. If disabled, messages are only saved when you exit the chat (default: `true`)</dd>

  <dt>`sessions.retention_days`</dt>
  <dd>Number of days to retain sessions before automatic cleanup. Set to `0` to keep sessions indefinitely (default: `30`)</dd>
</dl>

## Auto-Compact: Extended Conversations

Infrastructure conversations often span days or weeks -- VPC migrations, security audits, architecture
reviews. Without auto-compact, sessions hit message limits and start losing early context. Auto-compact
solves this by summarizing older messages instead of dropping them, so sessions can continue indefinitely.

### How It Works

When a session reaches its trigger threshold (default: 75% of `max_history_messages`), auto-compact
summarizes the oldest messages (default: 40%) into a concise summary that preserves infrastructure
decisions, technical details, and security context. The AI then sees the summary plus recent messages,
keeping full awareness of the conversation while using far fewer tokens.

<Terminal title="Example: Auto-Compact in Action">
```bash
# Session has 50-message limit, reaches 38 messages (75%)
# Compaction summarizes messages 1-20, keeps messages 21-50 verbatim

You: "What CIDR did we choose for production again?"

AI: "Based on our earlier discussion, we chose 10.0.0.0/16 for production
     to ensure adequate IP space for growth..."
     # AI still knows context from messages 1-20 via the summary
```
</Terminal>

When compaction triggers, you see status messages in the chat:

```
System: Compacting conversation (40 messages)...
System: Conversation compacted successfully (40 messages summarized, ~8000 tokens saved)
```

### Auto-Compact Configuration

<File title="atmos.yaml">
```yaml
settings:
  ai:
    max_history_messages: 50

    sessions:
      enabled: true
      auto_compact:
        enabled: true                     # Enable auto-compact (default: false)
        trigger_threshold: 0.75           # Compact at 75% capacity
        compact_ratio: 0.4                # Summarize oldest 40%
        preserve_recent: 10               # Always keep last 10 messages verbatim
        use_ai_summary: true              # Use AI for smart summaries
        summary_provider: "anthropic"     # Optional: provider for summaries
        summary_model: "claude-haiku-4-5-20251001"  # Optional: cheaper model
        summary_max_tokens: 2048          # Max tokens for summaries
        show_summary_markers: false       # Show [SUMMARY] tags for debugging
```
</File>

### Auto-Compact Options

<dl>
  <dt>`auto_compact.enabled`</dt>
  <dd>Enable intelligent auto-compaction. When `false`, uses traditional truncation (drops old messages). Default: `false`</dd>

  <dt>`auto_compact.trigger_threshold`</dt>
  <dd>Percentage (0.0-1.0) of `max_history_messages` at which to trigger compaction. Default: `0.75` (75%). Example: With 50-message limit, triggers at 38 messages.</dd>

  <dt>`auto_compact.compact_ratio`</dt>
  <dd>Percentage (0.0-1.0) of oldest messages to summarize. Default: `0.4` (40%). Example: With 50 messages, summarizes oldest 20.</dd>

  <dt>`auto_compact.preserve_recent`</dt>
  <dd>Number of most recent messages to never compact. Default: `10`. Ensures immediate context is always available verbatim.</dd>

  <dt>`auto_compact.use_ai_summary`</dt>
  <dd>Use AI to generate intelligent summaries. If `false`, uses simple concatenation. Default: `true` (recommended)</dd>

  <dt>`auto_compact.summary_provider`</dt>
  <dd>AI provider for generating summaries. Defaults to `default_provider`. Useful for using a cheaper provider to reduce costs.</dd>

  <dt>`auto_compact.summary_model`</dt>
  <dd>Model for summary generation. Defaults to provider's default model. Use faster/cheaper models like `claude-haiku-4-5-20251001` to optimize costs.</dd>

  <dt>`auto_compact.summary_max_tokens`</dt>
  <dd>Maximum tokens for generated summaries. Default: `2048`. Controls summary length and cost.</dd>

  <dt>`auto_compact.show_summary_markers`</dt>
  <dd>Display visual markers `[SUMMARY: Messages 1-20]` around summaries. Default: `false`. Enable for debugging.</dd>
</dl>

### What Gets Preserved vs. Dropped

Summaries preserve infrastructure decisions (CIDR choices, component selections, architectural
patterns), technical details (stack names, file paths, configuration values), security context
(security group rules, IAM policies, encryption settings), and validation results (errors found
and resolutions).

Conversational filler ("Thanks!", "OK"), step-by-step process narration, and tool execution
details are intentionally filtered out.

### Example Configurations

**Cost-optimized** -- use a cheaper model for summaries:

<File title="atmos.yaml">
```yaml
settings:
  ai:
    default_provider: "anthropic"
    providers:
      anthropic:
        model: "claude-sonnet-4-6"
        api_key: !env "ANTHROPIC_API_KEY"
    max_history_messages: 50
    sessions:
      auto_compact:
        enabled: true
        summary_model: "claude-haiku-4-5-20251001"
```
</File>

**Privacy-first** -- keep everything local with Ollama:

<File title="atmos.yaml">
```yaml
settings:
  ai:
    default_provider: "ollama"
    providers:
      ollama:
        model: "llama4"
        base_url: "http://localhost:11434/v1"
    max_history_messages: 100
    sessions:
      auto_compact:
        enabled: true
        summary_provider: "ollama"
```
</File>

**Aggressive compaction** -- for very long conversations (100+ messages):

<File title="atmos.yaml">
```yaml
settings:
  ai:
    max_history_messages: 100
    sessions:
      auto_compact:
        enabled: true
        trigger_threshold: 0.6
        compact_ratio: 0.5
        preserve_recent: 15
```
</File>

### Performance and Costs

| Metric | Value |
|--------|-------|
| Original messages (20) | ~8,000 tokens |
| AI-generated summary | ~300-500 tokens |
| Token reduction | ~94% |
| Break-even point | ~2 messages after compaction |

Auto-compact pays for itself quickly in long conversations. The summary generation costs one API
call (~8,000 input + 500 output tokens), but every subsequent message saves ~7,500 tokens in context.

## Using Sessions

### Starting and Resuming Sessions

<Terminal title="atmos ai chat --session">
```bash
# Start a named session (creates if new, resumes if existing)
atmos ai chat --session vpc-refactor

# Start an anonymous session (auto-generated name)
atmos ai chat
```
</Terminal>

### Managing Sessions in the TUI

The chat TUI provides markdown rendering, syntax highlighting, history navigation with arrow keys,
and multi-line input with `Shift+Enter`. See the [`atmos ai chat` documentation](/cli/commands/ai/chat)
for details on all interactive features.

#### Creating Sessions (Ctrl+N)

Press `Ctrl+N` to open the create session form. You can enter a custom name, select an AI provider,
and pick a model. Each session remembers which provider and model it was created with.

| Provider | Default Model | Type |
|----------|--------------|------|
| Anthropic (Claude) | `claude-sonnet-4-6` | Cloud |
| OpenAI (GPT) | `gpt-4o` | Cloud |
| Google (Gemini) | `gemini-2.5-flash` | Cloud |
| xAI (Grok) | `grok-4` | Cloud |
| Ollama | `llama4` | Local |
| AWS Bedrock | `anthropic.claude-sonnet-4-6` | Enterprise |
| Azure OpenAI | `gpt-4o` | Enterprise |

#### Session Picker (Ctrl+L)

Press `Ctrl+L` to open the session picker. It displays all sessions with their name, provider badge
(color-coded), creation date, and message count.

**Session picker controls:**

| Key | Action |
|-----|--------|
| `Up`/`Down` or `j`/`k` | Navigate sessions |
| `Enter` | Switch to selected session |
| `n` or `Ctrl+N` | Create a new session |
| `d` | Delete selected session (with confirmation) |
| `r` | Rename selected session (inline editing) |
| `f` | Cycle through provider filters |
| `Esc` or `q` | Return to current chat |

The `f` key cycles through provider filters (All, Claude, GPT, Gemini, Grok, Ollama, Bedrock,
Azure), letting you narrow down sessions by provider. All operations (select, delete, rename)
work with filters active.

:::warning Permanent Action
Deleting a session permanently removes all conversation history for that session. This action cannot be undone.
:::

### Skill Persistence

Sessions remember which AI skill you are using. When you switch skills during a conversation
(with `Ctrl+A`), that choice is automatically saved. When you resume the session later, it
continues using the same skill.

This lets you maintain multiple sessions with different specializations:

```
vpc-security-review      [atmos-validation]   # Validation specialist
component-refactor       [atmos-components]   # Component architecture
stack-analysis          [atmos-stacks]        # Stack investigation
terraform-planning      [atmos-terraform]     # Terraform orchestration
```

:::info Skill Availability
If you resume a session that was using a custom skill that is no longer available in your
configuration, Atmos falls back to the default skill and logs a debug message.
:::

### Listing and Cleaning Sessions

<Terminal title="atmos ai sessions">
```bash
# List all sessions with metadata
atmos ai sessions

# Clean sessions older than 30 days
atmos ai sessions clean --older-than 30d
```
</Terminal>

Supported time units for `--older-than`: `h` (hours), `d` (days), `w` (weeks), `m` (months).

## Session Storage

Sessions are stored locally in a SQLite database at `<project-root>/.atmos/sessions/sessions.db`.
The database is created automatically on first use and uses Write-Ahead Logging (WAL) for optimal
performance -- writes don't block reads, so your chat stays fast even with long histories.

Each session stores your messages, AI responses, session metadata (name, timestamps, model, provider),
and context references (stack files and components discussed). No API keys or credentials are stored.

### Backup and Restore

<Terminal title="shell">
```bash
# Backup
cp .atmos/sessions/sessions.db .atmos/sessions/sessions.db.backup

# Restore
cp .atmos/sessions/sessions.db.backup .atmos/sessions/sessions.db

# Export for sharing
sqlite3 .atmos/sessions/sessions.db .dump > session-export.sql
```
</Terminal>

## Privacy and Security

Session data is stored locally on your machine. No data is sent to AI providers except during active
conversations, when the message history is sent to provide context. API keys and credentials are
never stored in the session database.

If your conversations contain sensitive information (AWS account IDs, infrastructure details),
be aware that this data is sent to your AI provider when you resume a session.

To delete all session data:

<Terminal title="shell">
```bash
atmos ai sessions clean --older-than 0d

# Or remove the database entirely
rm .atmos/sessions/sessions.db
```
</Terminal>

## Best Practices

**Naming** -- Use descriptive session names that reflect the topic: `vpc-migration-2025`,
`rds-troubleshooting`, `security-review`. Avoid generic names like `test` or `temp`.

**Organization** -- Keep one topic per session. Use separate sessions for different concerns
(security, performance, cost). This also lets you assign different AI skills per session.

**Retention** -- Choose a retention period based on your workflow:

<File title="atmos.yaml">
```yaml
sessions:
  retention_days: 7    # Active development (keep recent only)
  retention_days: 90   # Long-term projects
  retention_days: 0    # Never auto-delete (manual cleanup only)
```
</File>

## Troubleshooting

| Problem | Solution |
|---------|----------|
| Session not found | Run `atmos ai sessions` to verify the name exists |
| Database corruption | Run `sqlite3 .atmos/sessions/sessions.db "PRAGMA integrity_check;"` or restore from backup |
| Disk space issues | Run `atmos ai sessions clean --older-than 7d` then `sqlite3 .atmos/sessions/sessions.db "VACUUM;"` |
| Permission errors | Run `chmod 755 .atmos/sessions && chmod 644 .atmos/sessions/sessions.db` |
