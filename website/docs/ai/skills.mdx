# AI Skills

<Experimental />

Atmos AI includes a powerful skill system that provides specialized AI assistants for different infrastructure tasks. Each skill has a customized system prompt and tool access tailored to its specific domain. Skills follow the [Agent Skills open standard](https://agentskills.io) governed by the Linux Foundation's Agentic AI Foundation (AAIF).

## Overview

Skills are specialized AI assistants that focus on specific aspects of infrastructure management. Instead of having one general-purpose AI, you can switch between skills optimized for different tasks like Terraform management, security auditing, stack analysis, configuration validation, and more.

### Why Use Skills?

1. **Focused Expertise**: Each skill has specialized knowledge and instructions
2. **Better Results**: Task-specific prompts produce more accurate outputs
3. **Tool Safety**: Skills can restrict access to dangerous operations
4. **Workflow Optimization**: Switch skills as you move through different phases of work

## Installing Skills

Atmos AI skills are distributed via the [Agent Skills marketplace](https://agentskills.io). The official Atmos skills package contains 21 specialized skills covering all aspects of infrastructure orchestration.

### Install All Atmos Skills

<Terminal title="shell">
```bash
# Install all 21 Atmos agent skills from the official package
atmos ai skill install cloudposse/atmos

# Install with a specific version
atmos ai skill install cloudposse/atmos@v1.200.0

# Skip confirmation prompt (for automation)
atmos ai skill install cloudposse/atmos --yes
```
</Terminal>

### Manage Installed Skills

<Terminal title="shell">
```bash
# List installed skills
atmos ai skill list

# Uninstall a specific skill
atmos ai skill uninstall atmos-terraform

# Force reinstall all skills
atmos ai skill install cloudposse/atmos --force
```
</Terminal>

### No Skills Installed

If no skills are installed, `atmos ai chat` works with a minimal fallback prompt and displays a hint to install specialized skills:

<Terminal title="atmos ai chat">
```bash
atmos ai chat
# Works with basic Atmos knowledge
# Hint: "Install specialized skills with 'atmos ai skill install'"
```
</Terminal>

## Skill Prompt System

Atmos AI uses a **SKILL.md format** that follows the [Agent Skills open standard](https://agentskills.io) governed by the Linux Foundation's Agentic AI Foundation (AAIF). Skills are installed from the marketplace and stored in `~/.atmos/skills/`.

### Architecture

**Separation of Concerns:**
- **Go code** (`pkg/ai/skills/`) - Skill infrastructure, tool execution, registry, session management
- **SKILL.md files** (`~/.atmos/skills/`) - Installed skill metadata and instructions (from marketplace)

This design provides several key benefits:

**For Development:**
- 🔧 Update skills without recompiling Atmos (just `atmos ai skill install --force`)
- 📝 Easy to maintain and version control prompt changes
- 🤝 Community can contribute skill improvements via pull requests
- 📊 Track prompt evolution in Git history
- 🧪 Test different prompt strategies easily

**For Users:**
- 🎯 Richer, more specialized skill responses
- 🔄 Consistent skill behavior across sessions
- 📚 Skills stay up-to-date with latest Atmos patterns
- ⚡ Token-efficient progressive disclosure (prompts load only when skill is activated)

### How It Works

#### 1. Marketplace-Installed Skills

Skills are installed from the marketplace to `~/.atmos/skills/`:

```
~/.atmos/skills/
├── cloudposse/atmos/
│   ├── atmos-terraform/
│   │   └── SKILL.md
│   ├── atmos-components/
│   │   └── SKILL.md
│   ├── atmos-config/
│   │   └── SKILL.md
│   ├── atmos-security/
│   │   └── SKILL.md
│   └── ... (21 skills total)
└── registry.json
```

**Key characteristics:**
- Skills are **installed separately** from the Atmos binary
- Update skills without upgrading Atmos: `atmos ai skill install --force`
- Multi-skill packages install all skills at once
- Each skill can reference additional files for richer prompts

#### 2. Dynamic Loading

Skills are loaded when the TUI initializes:

```
TUI Initialization
  → LoadInstalledSkills() from ~/.atmos/skills/
  → Load custom skills from atmos.yaml
  → Select default skill (or fallback)

Skill Switching (Ctrl+A)
  → User selects skill from installed list
  → Skill's system prompt is already loaded

No Skills Installed
  → Fallback skill provides basic Atmos assistance
  → Hint: "Install skills with 'atmos ai skill install'"
```

This **progressive disclosure** approach minimizes token usage - each skill's instructions are only loaded when that skill is active.

#### 3. SKILL.md File Structure

Each skill follows the SKILL.md format with YAML frontmatter and Markdown body:

```markdown
---
name: skill-name
display_name: "Skill Display Name"
description: Brief description of the skill
version: 1.0.0
category: general|analysis|refactor|security|validation|optimization
icon: 🤖

tools:
  allowed:
    - tool_name1
    - tool_name2
  restricted:
    - tool_name3
---

# Skill: [Name] [Icon]

## Role
Brief description of the skill's purpose and expertise

## Your Expertise
- Domain knowledge areas
- Specializations
- Key competencies

## Instructions
Step-by-step guidance for the skill to follow

## Best Practices
Extracted from Atmos documentation (llms.txt, llms-full.txt)

## Tools You Should Use
Recommended tools for this skill's domain

## Example Workflows
Common usage patterns and examples

## Response Style
How the skill should format and present information
```

**Example excerpt from a marketplace skill's SKILL.md:**

```markdown
---
name: atmos-stacks
display_name: "Stack Expert"
description: Analyze stack configurations and dependencies
version: 1.0.0
category: analysis

tools:
  allowed:
```

#### 4. Content Sources

Skill prompts are derived from:

| Source | Purpose |
|--------|---------|
| **llms.txt** | Atmos documentation index and structure |
| **llms-full.txt** | Comprehensive Atmos knowledge base |
| **website/docs/** | Official documentation and examples |
| **Best Practices** | Proven patterns from production use |
| **Community Feedback** | User-contributed improvements |

Each skill's prompt extracts relevant sections from these sources. For example:

- **atmos-stacks** gets: Stack concepts, import patterns, `describe` commands
- **atmos-components** gets: Component design best practices, Terraform patterns
- **atmos-validation** gets: OPA/Rego policies, JSON Schema, schema manifests
- **atmos-config** gets: YAML structure, configuration merging, error messages

### Contributing to Skill Prompts

Want to improve a skill? Here's how:

#### View Current Prompts

Skill prompts are in the Atmos repository under `agent-skills/skills/`:

```bash
# Clone the repository
git clone https://github.com/cloudposse/atmos.git
cd atmos

# View skill prompts
ls -la agent-skills/skills/
cat agent-skills/skills/atmos-terraform/SKILL.md
```

#### Modify a Prompt

1. **Edit the SKILL.md file:**
   ```bash
   # Make your improvements
   vim agent-skills/skills/atmos-terraform/SKILL.md
   ```

2. **Test locally:**
   ```bash
   # Install your modified skills locally
   atmos ai skill install . --force

   # Test the skill
   atmos ai chat
   # Press Ctrl+A, select the modified skill
   # Verify the skill behaves as expected
   ```

3. **Submit a pull request:**
   ```bash
   git checkout -b improve-terraform-skill
   git add agent-skills/skills/atmos-terraform/SKILL.md
   git commit -m "Improve Terraform skill: add state migration guidance"
   git push origin improve-terraform-skill
   # Create PR on GitHub
   ```

#### Guidelines for Prompt Changes

When modifying skill prompts, follow these guidelines:

**DO:**
- ✅ Add specific, actionable instructions
- ✅ Include concrete examples and workflows
- ✅ Extract knowledge from official Atmos docs
- ✅ Test your changes with real scenarios
- ✅ Maintain the established structure (Role, Expertise, Instructions, etc.)

**DON'T:**
- ❌ Make prompts unnecessarily long (keep under 10KB)
- ❌ Duplicate content across multiple skills
- ❌ Add instructions unrelated to the skill's domain
- ❌ Remove important safety considerations
- ❌ Break the SKILL.md structure

**Example of a good improvement:**

```diff
# In atmos-validation/SKILL.md

## Instructions

+### Check for Unused Security Groups
+
+When auditing network security:
+1. List all security groups in the stack
+2. Check for groups with no attached resources
+3. Flag unused groups as cost/security risk
+4. Recommend cleanup workflow
+
 ### Issue 1: Overly Permissive IAM
 ...
```

## Skill Configuration Reference

### Skill Schema

```yaml
skills:
  <skill-name>:
    display_name: string      # User-facing name (required)
    description: string       # What this skill does (required)
    system_prompt: string     # Specialized instructions (required)
    allowed_tools: []         # Tool names (empty = all tools)
    restricted_tools: []      # Tools requiring confirmation
    category: string          # "analysis", "refactor", "security", etc.
```

### Available Tools

**Atmos Commands**:
- `atmos_describe_component` - Get component configuration
- `atmos_describe_affected` - See impact analysis
- `atmos_describe_stacks` - List and describe all stacks
- `atmos_describe_config` - Show Atmos configuration
- `atmos_list_stacks` - List available stacks
- `atmos_list_components` - List available components
- `atmos_validate_stacks` - Validate stack configurations
- `atmos_get_template_context` - Get template variable context

**File Operations**:
- `read_file` - Read any file
- `edit_file` - Edit file content
- `write_file` - Create new files
- `read_stack_file` - Read stack YAML files
- `read_component_file` - Read component files
- `list_component_files` - List files in component directory
- `search_files` - Search for content across files

**Execution**:
- `execute_bash` - Execute shell commands
- `execute_atmos_command` - Execute Atmos commands

**Validation**:
- `validate_file_lsp` - Validate files using LSP (if enabled)

### Tool Safety

**Allowed Tools** (empty = all):
```yaml
allowed_tools: []  # Skill can use any tool
```

**Restricted Tools** (require confirmation):
```yaml
allowed_tools:
  - read_file
  - edit_file
restricted_tools:
  - edit_file  # User must confirm before skill can edit files
```

**Read-Only Skill**:
```yaml
allowed_tools:
  - atmos_describe_component
  - atmos_list_stacks
  - read_file
  - search_files
# No write/execute tools = read-only
```

## Best Practices

### 1. Use the Right Skill for the Task

Don't use the general skill for everything:

❌ **Wrong**:
```bash
atmos ai chat
You: "Audit my infrastructure for security issues"
# General skill lacks security expertise
```

✅ **Right**:
```bash
atmos ai chat
# Press Ctrl+A, select "atmos-validation"
You: "Audit my infrastructure for security issues"
# Specialized skill provides comprehensive analysis
```

### 2. Switch Skills as Your Work Progresses

Different phases need different skills:

```bash
# Phase 1: Understanding (atmos-stacks)
Ctrl+A → atmos-stacks
"Explain the prod-use1-eks configuration"

# Phase 2: Improvement (atmos-components)
Ctrl+A → atmos-components
"Refactor the EKS component to use module v5"

# Phase 3: Validation (atmos-validation)
Ctrl+A → atmos-validation
"Validate all my changes"

# Phase 4: Terraform Review (atmos-terraform)
Ctrl+A → atmos-terraform
"Check the Terraform plan for any issues"
```

### 3. Create Domain-Specific Skills

For specialized workflows, create custom skills:

**Multi-Region Deployment Skill**:
```yaml
skills:
  multi-region:
    display_name: "Multi-Region Deployer"
    description: "Handles multi-region deployment workflows"
    system_prompt: |
      You specialize in multi-region deployments.

      ALWAYS consider:
      - Regional resource naming
      - Cross-region dependencies
      - Data replication
      - Failover configurations

      When deploying to multiple regions, ensure consistency.
```

**Migration Skill**:
```yaml
skills:
  migrator:
    display_name: "Infrastructure Migrator"
    description: "Migrates infrastructure between accounts/regions"
    system_prompt: |
      You specialize in infrastructure migrations.

      MIGRATION CHECKLIST:
      - Verify state file location
      - Check cross-account permissions
      - Plan for DNS cutover
      - Test rollback procedures

      Always provide step-by-step migration plans.
```

### 4. Restrict Dangerous Operations

For sensitive environments, restrict write access:

```yaml
skills:
  prod-analyzer:
    display_name: "Production Analyzer"
    description: "Read-only analysis of production infrastructure"
    allowed_tools:
      # Only read operations
      - atmos_describe_component
      - atmos_describe_stacks
      - atmos_list_stacks
      - read_file
      - read_stack_file
      - search_files
      # No edit_file, write_file, or execute_bash
```

### 5. Document Custom Skill Purpose

Add clear descriptions so team members know when to use each skill:

```yaml
skills:
  disaster-recovery:
    display_name: "Disaster Recovery Planner"
    description: "Creates and reviews DR plans for infrastructure. Use for: backup configs, failover testing, RTO/RPO analysis."
    # Clear description tells users exactly when to use this skill
```

## Comparison with Claude Code Subagents

Atmos has two types of AI assistants serving different purposes:

| Feature | Atmos AI Skills (this page) | [Claude Code Integration](/ai/claude-code-integration) |
|---------|---------------------------|--------------------------|
| **Location** | Inside Atmos AI | External (IDE) |
| **Activation** | Ctrl+A in `atmos ai chat` | `@atmos-expert` in IDE |
| **Purpose** | Task specialization | Development integration |
| **Tool Access** | Direct Atmos tools | Via MCP protocol |
| **Configuration** | `atmos.yaml` skills section | `.claude/agents/*.md` |
| **Use Case** | Operations workflows | Code development |

**Use Atmos AI Skills when**:
- Working in terminal/CLI
- Need task-specific AI behavior
- Want tool access control
- Performing operations tasks

**Use Claude Code Integration when**:
- Working in IDE
- Writing/editing code
- Need IDE integration
- Performing development tasks

Both systems can work together! Use Atmos AI skills for infrastructure analysis in the terminal, and Claude Code subagents for code development in your IDE.

### Custom vs Marketplace Skills: Performance Implications

#### Marketplace Skills (Recommended)

**Advantages:**
- ✅ **Pre-optimized** - System prompts are carefully tuned for Atmos workflows
- ✅ **Consistent behavior** - Same experience across all Atmos installations
- ✅ **Maintained** - Updated with Atmos releases for improved performance
- ✅ **Well-tested** - Validated against common infrastructure patterns
- ✅ **Lower latency** - No configuration parsing overhead

**When to use:**
- You're performing standard infrastructure operations
- You want proven, reliable results
- Your team needs consistent AI behavior
- You're new to Atmos AI skills

#### Custom Skills

**Advantages:**
- ✅ **Domain-specific** - Tailored to your organization's needs
- ✅ **Workflow-optimized** - Built for your specific processes
- ✅ **Team knowledge** - Can encode institutional knowledge
- ✅ **Compliance-focused** - Can enforce company policies

**Considerations:**
- ⚠️ **Prompt engineering required** - Needs careful crafting for good results
- ⚠️ **Testing needed** - Must validate behavior across scenarios
- ⚠️ **Maintenance overhead** - Requires updates as workflows change
- ⚠️ **Inconsistent results** - Poorly-written prompts reduce quality

**When to create custom skills:**
```yaml
# Good reasons for custom skills
1. Specialized workflows
   Example: Multi-cloud deployment skill that understands your AWS+Azure patterns

2. Company-specific policies
   Example: Compliance skill that enforces internal naming conventions

3. Domain expertise
   Example: Cost optimization skill tuned for your infrastructure patterns

4. Team workflows
   Example: Migration skill with your specific migration checklist
```

**When to stick with marketplace skills:**
```yaml
# Better served by installed skills
❌ "I want a VPC analysis skill"
   → Use atmos-stacks - already optimized for stack analysis

❌ "I need to validate configs"
   → Use atmos-validation - knows OPA/Rego and JSON Schema

❌ "I need to refactor code"
   → Use atmos-components - understands IaC best practices

❌ "I need help with Terraform"
   → Use atmos-terraform - deep Terraform orchestration expertise
```

### Performance Optimization Tips

#### 1. Start Specific, Not General

❌ **Slower**:
```bash
# Using general skill for specialized task
atmos ai chat
You: "Check my infrastructure for security issues"
# Generic analysis, may miss domain-specific vulnerabilities
```

✅ **Faster**:
```bash
# Using specialized skill
atmos ai chat

# Focused, comprehensive validation analysis
```

✅ **Efficient**:
```bash
# Create session with appropriate skill
atmos ai chat --session vpc-review

# Perform reviews...

# Next time - skill automatically restored
atmos ai chat --session vpc-review
# ✅ Starts with atmos-validation, no need to switch
```

#### 3. Switch Skills During Workflows

✅ **Optimal workflow**:
```bash
# Analysis phase
Ctrl+A → atmos-stacks
"Analyze prod infrastructure"

# Improvement phase
Ctrl+A → atmos-components
"Refactor based on analysis"

# Validation phase
Ctrl+A → atmos-validation
"Validate changes"

"Review the Terraform plan"
```

Each skill provides better results for its phase than using General skill throughout.

```yaml
    description: "Does security, cost, compliance, and refactoring"
    system_prompt: |
      You do security AND cost AND compliance AND refactoring AND...
      # Too many responsibilities = poor performance in all areas
```

#### 5. Optimize Tool Access

**Give skills only what they need:**

✅ **Efficient - Minimal tool set**:
```yaml
skills:
  cost-analyzer:
    allowed_tools:
      - atmos_describe_component
      - atmos_list_stacks
      - read_file
    # Fast - AI only sees relevant tools
```

❌ **Slower - Unnecessary tools**:
```yaml
skills:
  cost-analyzer:
    allowed_tools: []  # All tools exposed
    # Slower - AI must evaluate all tools, even irrelevant ones
```

## Troubleshooting

### Skill Not Available

**Check configuration**:

<Terminal title="terminal">
```bash
atmos describe config | grep -A 20 "ai:"
```
</Terminal>

Ensure your custom skill is properly defined in `atmos.yaml`.

### Skill Not Using Specialized Prompt

**Verify skill is active**:

When you switch skills, you should see:
```
System: Switched to skill: atmos-stacks (Stack configuration, imports, inheritance, deep merging)
```

If you don't see this message, the skill switch didn't work. Press Ctrl+A again and ensure you press Enter on the desired skill.

### Custom Skill Tools Not Working

**Check tool names**:

Tool names must match exactly:
- ✅ `atmos_describe_component`
- ❌ `describe_component` (missing prefix)
- ❌ `atmos-describe-component` (wrong separator)

**Verify tool exists**:

<Terminal title="terminal">
```bash
# In chat, ask:
"What tools do you have access to?"

# Skill will list available tools
```
</Terminal>

### Restricted Tools Not Asking for Confirmation

This is expected behavior in the current implementation. The `restricted_tools` field is defined in the schema but confirmation prompts will be added in a future release. For now, use `allowed_tools` to completely block access to dangerous operations.
