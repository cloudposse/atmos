# AI Skills

<Experimental />

Atmos AI includes a powerful skill system that provides specialized AI assistants for different infrastructure tasks. Each skill has a customized system prompt and tool access tailored to its specific domain. Skills follow the [Agent Skills open standard](https://agentskills.io) governed by the Linux Foundation's Agentic AI Foundation (AAIF).

## Overview

Skills are specialized AI assistants that focus on specific aspects of infrastructure management. Instead of having one general-purpose AI, you can switch between skills optimized for:

- **Stack Analysis** - Understanding configurations and dependencies
- **Component Refactoring** - Improving Terraform/Helmfile code
- **Security Auditing** - Finding security vulnerabilities
- **Configuration Validation** - Ensuring YAML correctness
- **General Operations** - All-purpose assistance

### Why Use Skills?

1. **Focused Expertise**: Each skill has specialized knowledge and instructions
2. **Better Results**: Task-specific prompts produce more accurate outputs
3. **Tool Safety**: Skills can restrict access to dangerous operations
4. **Workflow Optimization**: Switch skills as you move through different phases of work

## Built-in Skills

- General questions about Atmos

- Ad-hoc exploration and troubleshooting

**Key Features**:
- No tool restrictions
- Balanced system prompt for general assistance
- Good starting point for any task

## Skill Prompt System

Atmos AI uses a **SKILL.md format** that follows the [Agent Skills open standard](https://agentskills.io) governed by the Linux Foundation's Agentic AI Foundation (AAIF). Each built-in skill has a dedicated SKILL.md file containing YAML frontmatter metadata and Markdown system prompt.

### Architecture

**Separation of Concerns:**
- **Go code** (`pkg/ai/skills/`) - Skill infrastructure, tool execution, registry, session management
- **SKILL.md files** (`pkg/ai/skills/builtin/`) - Skill metadata and instructions

This design provides several key benefits:

**For Development:**
- üîß Update skill behavior without recompiling Atmos
- üìù Easy to maintain and version control prompt changes
- ü§ù Community can contribute skill improvements via pull requests
- üìä Track prompt evolution in Git history
- üß™ Test different prompt strategies easily

**For Users:**
- üéØ Richer, more specialized skill responses
- üîÑ Consistent skill behavior across sessions
- üìö Skills stay up-to-date with latest Atmos patterns
- ‚ö° Token-efficient progressive disclosure (prompts load only when skill is activated)

### How It Works

#### 1. Embedded Filesystem

All skill SKILL.md files are embedded into the Atmos binary using Go's `//go:embed` directive:

```
pkg/ai/skills/builtin/
‚îú‚îÄ‚îÄ general/
‚îÇ   ‚îî‚îÄ‚îÄ SKILL.md                  # ü§ñ General skill (~6KB)
‚îú‚îÄ‚îÄ stack-analyzer/
‚îÇ   ‚îî‚îÄ‚îÄ SKILL.md                  # üìä Stack analysis (~6KB)
‚îú‚îÄ‚îÄ component-refactor/
‚îÇ   ‚îî‚îÄ‚îÄ SKILL.md                  # üîß Component refactoring (~6KB)
‚îú‚îÄ‚îÄ security-auditor/
‚îÇ   ‚îî‚îÄ‚îÄ SKILL.md                  # üîí Security auditing (~6KB)
‚îî‚îÄ‚îÄ config-validator/
    ‚îî‚îÄ‚îÄ SKILL.md                  # ‚úÖ Config validation (~6KB)
```

**Key characteristics:**
- Skills are **compiled into the binary** (no external files needed at runtime)
- Source files remain **editable** in the repository
- Changes require **rebuilding Atmos** to take effect
- Total size: ~30KB for all 5 skills

#### 2. Dynamic Loading

Prompts are loaded dynamically when you interact with skills:

```
Skill Initialization (startup)
  ‚Üí LoadSystemPrompt("general/SKILL.md")
  ‚Üí Skill has full instructions

Skill Switching (Ctrl+A)
  ‚Üí User selects "Stack Analyzer"
  ‚Üí LoadSystemPrompt("stack-analyzer/SKILL.md")
  ‚Üí Skill now has specialized stack analysis knowledge

Session Restore
  ‚Üí Load saved session
  ‚Üí LoadSystemPrompt("{saved-skill}/SKILL.md")
  ‚Üí Continue with same skill from last session
```

This **progressive disclosure** approach minimizes token usage - each skill's instructions are only loaded when that skill is active.

#### 4. Content Sources

Skill prompts are derived from:

| Source | Purpose |
|--------|---------|
| **llms.txt** | Atmos documentation index and structure |
| **llms-full.txt** | Comprehensive Atmos knowledge base |
| **website/docs/** | Official documentation and examples |
| **Best Practices** | Proven patterns from production use |

Each skill's prompt extracts relevant sections from these sources. For example:

- **Stack Analyzer** gets: Stack concepts, import patterns, `describe` commands
- **Component Refactor** gets: Component design best practices, Terraform patterns
- **Security Auditor** gets: Security checklist, IAM policies, encryption patterns
- **Config Validator** gets: YAML syntax, schema validation, error messages

## Skill Configuration Reference

### Skill Schema

```yaml
skills:
  <skill-name>:
    display_name: string      # User-facing name (required)
    description: string       # What this skill does (required)
    system_prompt: string     # Specialized instructions (required)
    allowed_tools: []         # Tool names (empty = all tools)
    restricted_tools: []      # Tools requiring confirmation
    category: string          # "analysis", "refactor", "security", etc.
```

### Available Tools

**Atmos Commands**:
- `atmos_describe_component` - Get component configuration
- `atmos_describe_affected` - See impact analysis
- `atmos_describe_stacks` - List and describe all stacks
- `atmos_describe_config` - Show Atmos configuration
- `atmos_list_stacks` - List available stacks
- `atmos_list_components` - List available components
- `atmos_validate_stacks` - Validate stack configurations
- `atmos_get_template_context` - Get template variable context

**File Operations**:
- `read_file` - Read any file
- `edit_file` - Edit file content
- `write_file` - Create new files
- `read_stack_file` - Read stack YAML files
- `read_component_file` - Read component files
- `list_component_files` - List files in component directory
- `search_files` - Search for content across files

**Execution**:
- `execute_bash` - Execute shell commands
- `execute_atmos_command` - Execute Atmos commands

**Validation**:
- `validate_file_lsp` - Validate files using LSP (if enabled)

### Tool Safety

**Allowed Tools** (empty = all):
```yaml
allowed_tools: []  # Skill can use any tool
```

**Restricted Tools** (require confirmation):
```yaml
allowed_tools:
  - read_file
  - edit_file
restricted_tools:
  - edit_file  # User must confirm before skill can edit files
```

**Read-Only Skill**:
```yaml
allowed_tools:
  - atmos_describe_component
  - atmos_list_stacks
  - read_file
  - search_files
# No write/execute tools = read-only
```

## Best Practices

### 1. Use the Right Skill for the Task

Don't use the general skill for everything:

‚ùå **Wrong**:
```bash
atmos ai chat
You: "Audit my infrastructure for security issues"
# General skill lacks security expertise
```

‚úÖ **Right**:
```bash
atmos ai chat
# Press Ctrl+A, select "Security Auditor"
You: "Audit my infrastructure for security issues"
# Security Auditor provides comprehensive security analysis
```

### 2. Switch Skills as Your Work Progresses

Different phases need different skills:

```bash
# Phase 1: Understanding (Stack Analyzer)
Ctrl+A ‚Üí Stack Analyzer
"Explain the prod-use1-eks configuration"

# Phase 2: Improvement (Component Refactor)
Ctrl+A ‚Üí Component Refactor
"Refactor the EKS component to use module v5"

# Phase 3: Validation (Config Validator)
Ctrl+A ‚Üí Config Validator
"Validate all my changes"

# Phase 4: Security Review (Security Auditor)
Ctrl+A ‚Üí Security Auditor
"Check for any security issues introduced"
```

### 3. Create Domain-Specific Skills

For specialized workflows, create custom skills:

**Multi-Region Deployment Skill**:
```yaml
skills:
  multi-region:
    display_name: "Multi-Region Deployer"
    description: "Handles multi-region deployment workflows"
    system_prompt: |
      You specialize in multi-region deployments.

      ALWAYS consider:
      - Regional resource naming
      - Cross-region dependencies
      - Data replication
      - Failover configurations

      When deploying to multiple regions, ensure consistency.
```

**Migration Skill**:
```yaml
skills:
  migrator:
    display_name: "Infrastructure Migrator"
    description: "Migrates infrastructure between accounts/regions"
    system_prompt: |
      You specialize in infrastructure migrations.

      MIGRATION CHECKLIST:
      - Verify state file location
      - Check cross-account permissions
      - Plan for DNS cutover
      - Test rollback procedures

      Always provide step-by-step migration plans.
```

### 4. Restrict Dangerous Operations

For sensitive environments, restrict write access:

```yaml
skills:
  prod-analyzer:
    display_name: "Production Analyzer"
    description: "Read-only analysis of production infrastructure"
    allowed_tools:
      # Only read operations
      - atmos_describe_component
      - atmos_describe_stacks
      - atmos_list_stacks
      - read_file
      - read_stack_file
      - search_files
      # No edit_file, write_file, or execute_bash
```

### 5. Document Custom Skill Purpose

Add clear descriptions so team members know when to use each skill:

```yaml
skills:
  disaster-recovery:
    display_name: "Disaster Recovery Planner"
    description: "Creates and reviews DR plans for infrastructure. Use for: backup configs, failover testing, RTO/RPO analysis."
    # Clear description tells users exactly when to use this skill
```

### Custom vs Built-in Skills: Performance Implications

#### Built-in Skills (Recommended)

**Advantages:**
- ‚úÖ **Pre-optimized** - System prompts are carefully tuned for Atmos workflows
- ‚úÖ **Consistent behavior** - Same experience across all Atmos installations
- ‚úÖ **Maintained** - Updated with Atmos releases for improved performance
- ‚úÖ **Well-tested** - Validated against common infrastructure patterns
- ‚úÖ **Lower latency** - No configuration parsing overhead

**When to use:**
- You're performing standard infrastructure operations
- You want proven, reliable results
- Your team needs consistent AI behavior
- You're new to Atmos AI skills

#### Custom Skills

**Advantages:**
- ‚úÖ **Domain-specific** - Tailored to your organization's needs
- ‚úÖ **Workflow-optimized** - Built for your specific processes
- ‚úÖ **Team knowledge** - Can encode institutional knowledge
- ‚úÖ **Compliance-focused** - Can enforce company policies

**Considerations:**
- ‚ö†Ô∏è **Prompt engineering required** - Needs careful crafting for good results
- ‚ö†Ô∏è **Testing needed** - Must validate behavior across scenarios
- ‚ö†Ô∏è **Maintenance overhead** - Requires updates as workflows change
- ‚ö†Ô∏è **Inconsistent results** - Poorly-written prompts reduce quality

**When to create custom skills:**
```yaml
# Good reasons for custom skills
1. Specialized workflows
   Example: Multi-cloud deployment skill that understands your AWS+Azure patterns

2. Company-specific policies
   Example: Compliance skill that enforces internal naming conventions

3. Domain expertise
   Example: Cost optimization skill tuned for your infrastructure patterns

4. Team workflows
   Example: Migration skill with your specific migration checklist
```

**When to stick with built-in:**
```yaml
# Better served by built-in skills
‚ùå "I want a VPC analysis skill"
   ‚Üí Use Stack Analyzer - already optimized for this

‚ùå "I need a security checker"
   ‚Üí Use Security Auditor - comprehensive coverage built-in

‚ùå "I want to validate configs"
   ‚Üí Use Config Validator - knows Atmos schema perfectly

‚ùå "I need to refactor code"
   ‚Üí Use Component Refactor - understands IaC best practices
```

### Performance Optimization Tips

#### 1. Start Specific, Not General

‚ùå **Slower**:
```bash
# Using general skill for specialized task
atmos ai chat
You: "Check my infrastructure for security issues"
# Generic analysis, may miss domain-specific vulnerabilities
```

‚úÖ **Faster**:
```bash
# Using specialized skill
atmos ai chat
Ctrl+A ‚Üí Security Auditor
You: "Check my infrastructure for security issues"
# Focused, comprehensive security analysis
```

#### 2. Use Skill Persistence

‚úÖ **Efficient**:
```bash
# Create session with appropriate skill
atmos ai chat --session vpc-security-review
Ctrl+A ‚Üí Security Auditor
# Perform reviews...

# Next time - skill automatically restored
atmos ai chat --session vpc-security-review
# ‚úÖ Starts with Security Auditor, no need to switch
```

#### 3. Switch Skills During Workflows

‚úÖ **Optimal workflow**:
```bash
# Analysis phase
Ctrl+A ‚Üí Stack Analyzer
"Analyze prod infrastructure"

# Improvement phase
Ctrl+A ‚Üí Component Refactor
"Refactor based on analysis"

# Validation phase
Ctrl+A ‚Üí Config Validator
"Validate changes"

# Security review
Ctrl+A ‚Üí Security Auditor
"Security audit of changes"
```

Each skill provides better results for its phase than using General skill throughout.

#### 4. Limit Custom Skill Complexity

**Simple, focused skills perform better:**

‚úÖ **Good - Focused custom skill**:
```yaml
skills:
  cost-optimizer:
    description: "Analyzes and recommends cost optimizations"
    system_prompt: |
      You specialize in AWS cost optimization.
      Focus on: unused resources, rightsizing, reserved instances.
```

‚ùå **Bad - Overly complex skill**:
```yaml
skills:
  everything-skill:
    description: "Does security, cost, compliance, and refactoring"
    system_prompt: |
      You do security AND cost AND compliance AND refactoring AND...
      # Too many responsibilities = poor performance in all areas
```

#### 5. Optimize Tool Access

**Give skills only what they need:**

‚úÖ **Efficient - Minimal tool set**:
```yaml
skills:
  cost-analyzer:
    allowed_tools:
      - atmos_describe_component
      - atmos_list_stacks
      - read_file
    # Fast - AI only sees relevant tools
```

‚ùå **Slower - Unnecessary tools**:
```yaml
skills:
  cost-analyzer:
    allowed_tools: []  # All tools exposed
    # Slower - AI must evaluate all tools, even irrelevant ones
```

## Troubleshooting

### Skill Not Available

**Check configuration**:

<Terminal title="terminal">
```bash
atmos describe config | grep -A 20 "ai:"
```
</Terminal>

Ensure your custom skill is properly defined in `atmos.yaml`.

### Skill Not Using Specialized Prompt

**Verify skill is active**:

When you switch skills, you should see:
```
System: Switched to skill: Stack Analyzer (Analyze stack configurations and dependencies)
```

If you don't see this message, the skill switch didn't work. Press Ctrl+A again and ensure you press Enter on the desired skill.

### Custom Skill Tools Not Working

**Check tool names**:

Tool names must match exactly:
- ‚úÖ `atmos_describe_component`
- ‚ùå `describe_component` (missing prefix)
- ‚ùå `atmos-describe-component` (wrong separator)

**Verify tool exists**:

<Terminal title="terminal">
```bash
# In chat, ask:
"What tools do you have access to?"

# Skill will list available tools
```
</Terminal>

### Restricted Tools Not Asking for Confirmation

This is expected behavior in the current implementation. The `restricted_tools` field is defined in the schema but confirmation prompts will be added in a future release. For now, use `allowed_tools` to completely block access to dangerous operations.
