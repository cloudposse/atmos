<Intro>
The Atmos MCP Server exposes Atmos AI tools through the [Model Context Protocol](https://modelcontextprotocol.io) (MCP),
an open standard that lets any compatible AI client -- Claude Desktop, Claude Code, VS Code, Cursor, and
[many others](https://modelcontextprotocol.io/clients) -- connect to your infrastructure tools without custom integrations.
</Intro>

<Experimental />

**See also:** [MCP Command Reference](/cli/commands/mcp/start) for flag details and transport modes |
[MCP Configuration](/cli/configuration/mcp) for tool and permission settings in `atmos.yaml`

:::danger Security Warning
The MCP server has no built-in authentication or encryption.

- **stdio transport** (default) is safe for local use -- it runs as a subprocess with no network exposure.
- **HTTP transport** exposes your infrastructure data to anyone who can reach the port. Always add firewall rules, VPN, or a reverse proxy with auth before exposing it on a network.

See [Security Considerations](#security-considerations) for detailed guidance.
:::

## Quick Start

This walkthrough gets you from zero to querying stacks in Claude Desktop.

**Prerequisites:** Atmos v1.63.0+ and [Claude Desktop](https://claude.ai/download) v0.7.0+.

### 1. Enable AI in atmos.yaml

<File title="atmos.yaml">
```yaml
settings:
  ai:
    enabled: true
    tools:
      enabled: true
      require_confirmation: false  # optional, smoother experience
```
</File>

### 2. Configure Claude Desktop

Create or edit the config file at `~/.config/claude/claude_desktop_config.json` (macOS/Linux)
or `%APPDATA%\Claude\claude_desktop_config.json` (Windows):

<File title="claude_desktop_config.json">
```json
{
  "mcpServers": {
    "atmos": {
      "command": "atmos",
      "args": ["mcp", "start"],
      "cwd": "/path/to/your/atmos/project"
    }
  }
}
```
</File>

:::tip Working Directory
Set `cwd` to the directory containing your `atmos.yaml`. On Windows, use `atmos.exe` as the command
and double backslashes for paths (e.g., `C:\\Users\\You\\project`).
:::

### 3. Restart and Verify

Completely quit Claude Desktop and reopen it. After a few seconds, look for the MCP indicator icon in the
bottom-right corner. Click it to confirm "atmos" is listed with a green dot. Then try asking:

```
List all my Atmos stacks
```

Claude will call `atmos_list_stacks` and return your actual stack names.

If something goes wrong, check the [Troubleshooting](#troubleshooting) section.

## Available Tools

When connected, the MCP server exposes all enabled Atmos AI tools. For full details on each tool, its parameters,
and the permission system, see [AI Tools Configuration](/cli/configuration/ai/tools).

<dl>
  <dt>`atmos_describe_component`</dt>
  <dd>Get detailed component configuration for a stack.</dd>

  <dt>`atmos_list_stacks`</dt>
  <dd>List and filter stacks.</dd>

  <dt>`atmos_validate_stacks`</dt>
  <dd>Validate stack configurations.</dd>

  <dt>`atmos_terraform_plan`</dt>
  <dd>Generate Terraform plans.</dd>

  <dt>`atmos_read_component_file`</dt>
  <dd>Read a component source file.</dd>

  <dt>`atmos_read_stack_file`</dt>
  <dd>Read a stack configuration file.</dd>

  <dt>`atmos_write_component_file`</dt>
  <dd>Modify a component file (requires permission).</dd>

  <dt>`atmos_write_stack_file`</dt>
  <dd>Modify a stack file (requires permission).</dd>
</dl>

All tools respect your configured [permission settings](/cli/configuration/ai/tools).

Every client uses the same underlying command (`atmos mcp start`); only the config format differs.

## Security Considerations

### stdio Transport

Runs as a local subprocess under your user account with no network exposure. This is the safest option
for everyday use.

### HTTP Transport

Has no authentication or encryption by default. For any deployment beyond `localhost`:

<dl>
  <dt>Network isolation</dt>
  <dd>Use firewall rules, private networks, or VPN. Bind to `localhost` for single-user scenarios.</dd>

  <dt>Reverse proxy</dt>
  <dd>Put nginx or Caddy in front with TLS and authentication. See the <a href="#reverse-proxy-with-authentication">nginx example</a> below.</dd>

  <dt>SSH tunneling</dt>
  <dd>Forward the port over SSH for secure remote access: `ssh -L 8080:localhost:8080 user@server`.</dd>

  <dt>Tool permissions</dt>
  <dd>Restrict tools in `atmos.yaml` using `allowed_tools` and `blocked_tools`. Disable write operations if not needed.</dd>

  <dt>Monitoring</dt>
  <dd>Poll `/health` and log all tool executions.</dd>
</dl>

## Advanced Deployments

### Docker

<File title="Dockerfile">
```dockerfile
FROM cloudposse/atmos:latest

COPY atmos.yaml /atmos/atmos.yaml
COPY stacks/ /atmos/stacks/
COPY components/ /atmos/components/

WORKDIR /atmos
EXPOSE 8080

# Binding to 0.0.0.0 inside a container is safe; Docker port mapping controls external access.
CMD ["atmos", "mcp", "start", "--transport", "http", "--host", "0.0.0.0", "--port", "8080"]
```
</File>

<Terminal title="docker">
```bash
docker build -t atmos-mcp .
docker run -p 8080:8080 -e ANTHROPIC_API_KEY="${ANTHROPIC_API_KEY}" atmos-mcp
```
</Terminal>

### Kubernetes

<File title="deployment.yaml">
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: atmos-mcp-server
spec:
  replicas: 1
  selector:
    matchLabels:
      app: atmos-mcp
  template:
    metadata:
      labels:
        app: atmos-mcp
    spec:
      containers:
      - name: atmos-mcp
        image: atmos-mcp:latest
        ports:
        - containerPort: 8080
        env:
        - name: ANTHROPIC_API_KEY
          valueFrom:
            secretKeyRef:
              name: ai-credentials
              key: anthropic-api-key
        livenessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 30
---
apiVersion: v1
kind: Service
metadata:
  name: atmos-mcp-service
spec:
  selector:
    app: atmos-mcp
  ports:
  - port: 8080
    targetPort: 8080
```
</File>

### Reverse Proxy with Authentication

Example nginx config with TLS and basic auth:

<File title="nginx.conf">
```nginx
upstream atmos_mcp {
    server localhost:8080;
}

server {
    listen 443 ssl http2;
    server_name atmos-mcp.example.com;

    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    auth_basic "Atmos MCP Server";
    auth_basic_user_file /etc/nginx/.htpasswd;

    location / {
        proxy_pass http://atmos_mcp;
        proxy_http_version 1.1;
        proxy_set_header Connection '';
        proxy_buffering off;
        proxy_cache off;
    }

    location /sse {
        proxy_pass http://atmos_mcp/sse;
        proxy_http_version 1.1;
        proxy_set_header Connection '';
        proxy_set_header Cache-Control 'no-cache';
        proxy_buffering off;
        chunked_transfer_encoding on;
    }
}
```
</File>

## Troubleshooting

### stdio Connection Issues

**Server not appearing in Claude Desktop:**

1. Verify the config file location and JSON syntax. Use the paths listed in [Quick Start](#2-configure-claude-desktop).
2. Fully quit Claude Desktop (Cmd+Q / Alt+F4), wait a few seconds, and reopen.
3. Confirm `atmos` is in your PATH:

<Terminal title="verify">
```bash
which atmos
atmos mcp start   # should start without errors; Ctrl+C to exit
```
</Terminal>

4. Check Claude Desktop logs for errors mentioning "atmos":
   - macOS: `~/Library/Logs/Claude/mcp*.log`
   - Windows: `%APPDATA%\Claude\logs\mcp*.log`
   - Linux: `~/.config/Claude/logs/mcp*.log`

**Tools not appearing in conversations:**

Make sure `settings.ai.tools.enabled` is `true` in `atmos.yaml` and that you have not added the tools
to `blocked_tools`. If tools still do not appear, try asking Claude explicitly:
"Use the atmos_list_stacks tool to show my stacks."

**"Command not found" errors:**

Use the full path to `atmos` in your config. Run `which atmos` to find it, then set
`"command": "/usr/local/bin/atmos"` (or wherever it is installed).

### HTTP Connection Issues

<Terminal title="debug http">
```bash
# Check if the server is running
curl http://localhost:8080/health

# Check if the port is in use
lsof -i :8080

# Try a different port
atmos mcp start --transport http --port 3000
```
</Terminal>

If a firewall is blocking connections, allow the port (`sudo ufw allow 8080/tcp` on Linux,
or add `atmos` to the macOS application firewall).

### Common Configuration Errors

**"AI tools are not enabled"** -- Set `settings.ai.enabled: true` and `settings.ai.tools.enabled: true`
in `atmos.yaml`.

**Tools not executing** -- Check `allowed_tools` and `blocked_tools` in your AI tool config. Remove overly
restrictive entries.

**Slow responses** -- Increase `settings.ai.timeout_seconds` (default varies by provider). For HTTP transport,
check network latency. Monitor resource usage via `/health`.
