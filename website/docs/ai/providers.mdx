# AI Providers

<Experimental />

Atmos supports seven AI providers: cloud APIs for convenience, local models for privacy, and enterprise solutions for compliance. You can configure multiple providers and switch between them at any time.

## Supported Providers

| Provider | Default Model | Authentication | Best For |
|----------|---------------|----------------|----------|
| **Anthropic (Claude)** | `claude-sonnet-4-5-20250929` | API Key | Advanced reasoning, default choice |
| **OpenAI (GPT)** | `gpt-4o` | API Key | Widely available, strong general capabilities |
| **Google (Gemini)** | `gemini-2.5-flash` | API Key | Fast responses, larger context window |
| **xAI (Grok)** | `grok-4-latest` | API Key | OpenAI-compatible, real-time knowledge |
| **Ollama (Local)** | `llama3.3:70b` | None (local) | Privacy, offline use, zero API costs |
| **AWS Bedrock** | `anthropic.claude-sonnet-4-5-20250929-v1:0` | AWS Credentials | Enterprise security, AWS integration |
| **Azure OpenAI** | `gpt-4o` | API Key / Azure AD | Enterprise compliance, Azure integration |

## Provider Comparison

| Feature | Cloud (Claude, GPT, Gemini, Grok) | Ollama (Local) | Enterprise (Bedrock, Azure) |
|---------|-----------------------------------|----------------|------------------------------|
| **Setup** | Easy (API key) | Medium (install + download model) | Complex (cloud config) |
| **Cost** | Per-token pricing | Free | Per-token + cloud costs |
| **Data Privacy** | Sent to provider | 100% local | Stays in your cloud |
| **Offline** | No | Yes | No |
| **Rate Limits** | Yes | No | Yes (configurable) |
| **Compliance** | Provider-dependent | Your control | SOC2, HIPAA, ISO |
| **Private Network** | No | Yes | Yes (VPC/VNet) |

### Performance and Cost

| Provider | Context Window | Approx. Cost (per 1M tokens) |
|----------|----------------|------------------------------|
| **Anthropic (Claude)** | 200K tokens | $3-$15 (input/output) |
| **OpenAI (GPT-4o)** | 128K tokens | $2.50-$10 (input/output) |
| **Google (Gemini)** | up to 2M tokens | $0.075-$0.30 (input/output) |
| **xAI (Grok)** | 2M tokens | $3-$15 (input/output) |
| **Ollama** | 128K tokens | $0 (hardware only) |
| **AWS Bedrock** | 200K tokens | $3-$15 + AWS costs |
| **Azure OpenAI** | 128K tokens | $2.50-$10 + Azure costs |

:::tip Cost Optimization
For budget-conscious usage, try Gemini (cheapest cloud) or Ollama (free). For balanced cost and quality, Claude Sonnet or GPT-4o work well. Enterprise providers can leverage committed-use discounts.
:::

## Token Caching (Prompt Caching)

Token caching (prompt caching) reduces costs by reusing frequently-sent content like system prompts. Instead of paying full price for the same tokens every message, cached tokens cost significantly less.

| Provider | Discount | Auto/Manual | Configuration Required |
|----------|----------|-------------|----------------------|
| **Anthropic** | 90% | Manual markers | Yes (see below) |
| **OpenAI** | 50% | Automatic | None |
| **Gemini** | Free (included) | Automatic | None |
| **Grok** | 75% | Automatic | None |
| **Bedrock** | Up to 90% | Automatic | None |
| **Azure OpenAI** | 50-100% | Automatic | None |
| **Ollama** | N/A | N/A (local) | N/A |

Most providers implement caching automatically. Only Anthropic requires explicit configuration:

<File title="atmos.yaml">
```yaml
settings:
  ai:
    providers:
      anthropic:
        model: "claude-sonnet-4-5-20250929"
        api_key_env: "ANTHROPIC_API_KEY"
        max_tokens: 4096
        cache:
          enabled: true              # Enable prompt caching (default)
          cache_system_prompt: true  # Cache skill system prompt
          cache_project_instructions: true # Cache ATMOS.md content
```
</File>

The first message creates cache entries at normal cost. Subsequent messages reuse cached content at a 90% discount. Cache expires after 5 minutes of inactivity and refreshes automatically when you send another message.

To disable caching, set `cache.enabled: false`. To cache only the system prompt (not project instructions), set `cache_project_instructions: false`.

:::tip Maximize Cache Savings
For Anthropic, keep your ATMOS.md content stable during conversations and use named sessions for extended work on the same topic. For all other providers, caching works transparently with no action required.
:::

## Cloud Providers

Each cloud provider follows the same configuration pattern. Set your API key as an environment variable, then reference it in `atmos.yaml`.

### Anthropic (Claude)

<File title="atmos.yaml">
```yaml
providers:
  anthropic:
    model: "claude-sonnet-4-5-20250929"  # Recommended default
    # model: "claude-haiku-4-5-20251001"    # Fastest, most affordable
    # model: "claude-opus-4-1-20250805"     # Most capable reasoning
    api_key_env: "ANTHROPIC_API_KEY"
    max_tokens: 4096
```
</File>

<Terminal title="shell">
```bash
export ANTHROPIC_API_KEY="sk-ant-..."
```
</Terminal>

Get your API key at [console.anthropic.com](https://console.anthropic.com/).

:::tip Model Selection
Claude Sonnet 4.5 balances performance, speed, and cost for most workloads. Use Haiku 4.5 for high-volume tasks at lower cost, or Opus 4.1 for the highest accuracy on specialized reasoning.
:::

### OpenAI (GPT)

<File title="atmos.yaml">
```yaml
providers:
  openai:
    model: "gpt-4o"  # Recommended default
    # model: "o1"              # Flagship reasoning model
    # model: "o1-mini"         # Faster reasoning
    # model: "gpt-4o-mini"     # Fast and affordable
    api_key_env: "OPENAI_API_KEY"
    max_tokens: 4096
```
</File>

<Terminal title="shell">
```bash
export OPENAI_API_KEY="sk-..."
```
</Terminal>

Get your API key at [platform.openai.com/api-keys](https://platform.openai.com/api-keys).

:::info Max Tokens
OpenAI's newer models (o1, o1-mini) use `max_completion_tokens` internally. Atmos handles this automatically -- just set `max_tokens` in your configuration.
:::

### Google (Gemini)

<File title="atmos.yaml">
```yaml
providers:
  gemini:
    model: "gemini-2.5-flash"  # Recommended default
    # model: "gemini-2.5-pro"          # Best reasoning
    # model: "gemini-2.5-flash-lite"   # Fastest, cheapest
    api_key_env: "GEMINI_API_KEY"
    max_tokens: 8192
```
</File>

<Terminal title="shell">
```bash
export GEMINI_API_KEY="AI..."
```
</Terminal>

Get your API key at [aistudio.google.com/app/apikey](https://aistudio.google.com/app/apikey).

Gemini offers the most economical pricing among cloud providers while maintaining high quality, with context windows up to 2M tokens on some models.

### xAI (Grok)

<File title="atmos.yaml">
```yaml
providers:
  grok:
    model: "grok-4-latest"
    api_key_env: "XAI_API_KEY"
    max_tokens: 4096
    base_url: "https://api.x.ai/v1"
```
</File>

<Terminal title="shell">
```bash
export XAI_API_KEY="xai-..."
```
</Terminal>

Get your API key at [console.x.ai](https://console.x.ai).

:::warning Model Availability
As of 2025, only `grok-4-latest` is accessible via the xAI API. Other model IDs mentioned in xAI's documentation return "Model not found" errors.
:::

Grok offers a 2M token context window, built-in real-time search, and an OpenAI-compatible API. Note that it may exhibit slower response times (20+ seconds) with tool-heavy workloads compared to other providers.

## Local Provider

### Ollama

Ollama runs AI models entirely on your machine. No data leaves your computer, no API costs, no rate limits.

#### Installation

<Terminal title="shell">
```bash
# macOS
brew install ollama

# Linux
curl -fsSL https://ollama.com/install.sh | sh

# Windows: download from https://ollama.com/download
```
</Terminal>

Start the service and pull a model:

<Terminal title="shell">
```bash
ollama serve        # Start Ollama (Linux auto-starts as a service)
ollama pull llama3.3:70b   # Pull recommended model
ollama list                # List available models
```
</Terminal>

#### Model Selection

| Model | Size | RAM Required | Best For |
|-------|------|-------------|----------|
| `llama3.3:70b` | ~40GB | 64GB+ | Production use (recommended) |
| `llama3.1:8b` | ~5GB | 8GB+ | Quick queries, laptops |
| `llama3.2` | ~2GB | 4GB+ | Simple tasks only |
| `codellama:34b` | ~19GB | 32GB+ | Code generation |
| `codellama:13b` | ~8GB | 16GB+ | Code on mid-range hardware |
| `mixtral:8x7b` | ~26GB | 32GB+ | Complex reasoning |
| `mistral:7b` | ~4GB | 8GB+ | Fast responses, consumer hardware |

#### Configuration

**Local deployment (default):**

<File title="atmos.yaml">
```yaml
providers:
  ollama:
    model: "llama3.3:70b"
    # base_url defaults to http://localhost:11434/v1
    # No api_key_env needed for local instances
```
</File>

**Remote deployment** (offload storage and computation to a server):

<File title="atmos.yaml">
```yaml
providers:
  ollama:
    model: "llama3.3:70b"
    base_url: "http://<server-ip>:11434/v1"
    # Or with HTTPS:
    # base_url: "https://ollama.company.com/v1"
    # api_key_env: "OLLAMA_API_KEY"  # If behind auth proxy
```
</File>

To enable remote access on the server, set `OLLAMA_HOST=0.0.0.0:11434` before starting Ollama.

:::caution
Binding to `0.0.0.0` exposes the Ollama API to all network interfaces. Ensure the server is behind a firewall or reverse proxy with authentication in production environments.
:::

**Linux (systemd):**

<Terminal title="shell">
```bash
sudo systemctl edit ollama.service
# Add:
# [Service]
# Environment="OLLAMA_HOST=0.0.0.0:11434"

sudo systemctl daemon-reload
sudo systemctl restart ollama.service
```
</Terminal>

**macOS:**

<Terminal title="shell">
```bash
launchctl setenv OLLAMA_HOST "0.0.0.0:11434"
# Restart Ollama application
```
</Terminal>

**Docker:**

<Terminal title="shell">
```bash
docker run -d \
  --gpus=all \
  -v ollama:/root/.ollama \
  -p 11434:11434 \
  -e OLLAMA_HOST=0.0.0.0:11434 \
  --restart unless-stopped \
  --name ollama \
  ollama/ollama
```
</Terminal>

#### Cloud Deployment

You can deploy Ollama to any cloud provider (AWS, Azure, GCP) using VMs with optional GPU support, container services, or Kubernetes. The general approach is the same regardless of cloud:

1. Provision a VM or container with sufficient RAM (and GPU if desired)
2. Install Docker and Ollama (or use the `ollama/ollama` container image)
3. Set `OLLAMA_HOST=0.0.0.0:11434` for remote access
4. Open port 11434 in your firewall/security group
5. Pull models with `ollama pull <model>`
6. Point your Atmos config `base_url` at the server

:::warning Security for Remote Ollama
For production deployments, always put Ollama behind a reverse proxy (nginx, Caddy, Traefik) with HTTPS and authentication. Ollama has no built-in auth. Restrict CORS origins with `OLLAMA_ORIGINS`, use network isolation (VPC/VPN), and never expose directly to the public internet without auth.
:::

| Environment Variable | Purpose | Default |
|---------------------|---------|---------|
| `OLLAMA_HOST` | Server bind address | `127.0.0.1:11434` |
| `OLLAMA_ORIGINS` | CORS allowed origins | `*` (allow all) |
| `OLLAMA_MODELS` | Model storage path | `~/.ollama/models` |

#### Performance Tips

Hardware recommendations:
- **Apple Silicon (M1/M2/M3/M4):** Handles `llama3.3:70b` well with unified memory
- **16GB RAM:** Use `llama3.1:8b` or `mistral:7b`
- **32GB+ RAM:** Use `llama3.3:70b` or `mixtral:8x7b`
- **GPU acceleration** (NVIDIA/AMD/Apple Silicon) provides 10-50x faster inference; Ollama detects and uses it automatically

<File title="atmos.yaml">
```yaml
providers:
  ollama:
    model: "llama3.3:70b"
    max_tokens: 4096       # Smaller = faster
    timeout_seconds: 120   # Longer timeout for large models
```
</File>

#### Troubleshooting

<dl>
<dt>Connection Refused</dt>
<dd>Make sure Ollama is running (<code>curl http://localhost:11434/api/version</code>) and the port in your config matches.</dd>

<dt>Model Not Found</dt>
<dd>Pull the model first with <code>ollama pull &lt;model&gt;</code> and verify the name matches your config exactly (<code>ollama list</code>).</dd>

<dt>Slow Responses</dt>
<dd>Switch to a smaller model, reduce <code>max_tokens</code>, or check that GPU acceleration is active.</dd>

<dt>Out of Memory</dt>
<dd>Use a smaller model. For example, switch from <code>llama3.3:70b</code> to <code>llama3.1:8b</code>.</dd>
</dl>

## Enterprise Providers

Enterprise providers keep data within your own cloud infrastructure and support compliance requirements like HIPAA, SOC2, and ISO 27001.

### AWS Bedrock

Run Claude models through AWS infrastructure with IAM-based access, CloudTrail audit logging, and VPC endpoint support.

<File title="atmos.yaml">
```yaml
providers:
  bedrock:
    model: "anthropic.claude-sonnet-4-5-20250929-v1:0"
    # model: "anthropic.claude-haiku-4-5-20251001-v1:0"
    # model: "global.anthropic.claude-sonnet-4-5-20250929-v1:0"  # Global endpoint
    max_tokens: 4096
    base_url: "us-east-1"  # AWS region
```
</File>

Authentication uses the standard AWS SDK credential chain (IAM roles, profiles, environment variables):

<Terminal title="shell">
```bash
# Option 1: Environment variables
export AWS_ACCESS_KEY_ID="AKIA..."
export AWS_SECRET_ACCESS_KEY="..."

# Option 2: AWS Profile
export AWS_PROFILE="bedrock-profile"

# Option 3: IAM role (automatic in EC2/ECS/EKS)
```
</Terminal>

:::warning Bedrock Endpoints
Use the `global.` prefix for dynamic routing across regions, or the `anthropic.` prefix for a specific region. Cross-region inference profiles use IDs like `us.anthropic.claude-3-5-sonnet-20241022-v2:0`.
:::

### Azure OpenAI

Run GPT models through Azure infrastructure with Azure AD integration, private endpoints, and customer-managed encryption keys.

<File title="atmos.yaml">
```yaml
providers:
  azureopenai:
    model: "gpt-4o"  # Your Azure deployment name (NOT the OpenAI model name)
    base_url: "https://<your-resource>.openai.azure.com"  # Required
    api_version: "2024-02-15-preview"  # Required
    api_key_env: "AZURE_OPENAI_API_KEY"
    max_tokens: 4096
```
</File>

<Terminal title="shell">
```bash
export AZURE_OPENAI_API_KEY="..."
```
</Terminal>

:::warning Azure-Specific Configuration
The `model` field must be your **deployment name** from Azure Portal, not the OpenAI model name. Both `base_url` (your resource endpoint) and `api_version` are required. Like OpenAI, Azure's newer deployments (gpt-5, o1-preview) use `max_completion_tokens` internally -- Atmos handles this automatically. See [Azure OpenAI Documentation](https://learn.microsoft.com/en-us/azure/ai-services/openai/) for available API versions.
:::

## Multi-Provider Configuration

You can configure multiple providers and switch between them. Set `default_provider` for CLI commands, and press `Ctrl+P` during a chat session to switch mid-conversation.

<File title="atmos.yaml">
```yaml
settings:
  ai:
    enabled: true
    default_provider: "anthropic"

    providers:
      anthropic:
        model: "claude-sonnet-4-5-20250929"
        api_key_env: "ANTHROPIC_API_KEY"
        max_tokens: 4096

      openai:
        model: "gpt-4o"
        api_key_env: "OPENAI_API_KEY"
        max_tokens: 4096

      ollama:
        model: "llama3.3:70b"

      bedrock:
        model: "anthropic.claude-sonnet-4-5-20250929-v1:0"
        base_url: "us-east-1"

      azureopenai:
        model: "gpt-4o"
        api_key_env: "AZURE_OPENAI_API_KEY"
        base_url: "https://mycompany.openai.azure.com"
        api_version: "2024-02-15-preview"
```
</File>

### Provider Isolation

Each provider maintains its own completely isolated conversation history. When you switch providers, the new provider only sees messages from its own thread. When you switch back, it picks up right where you left off. UI notifications like "Switched to..." are never sent to any provider.

This means each provider has clean, relevant context without noise from other models, and you can easily compare how different providers respond to the same questions.

## Security Best Practices

<dl>
<dt>Cloud Providers (Anthropic, OpenAI, Google, xAI)</dt>
<dd>Use environment variables for API keys (never commit to git), rotate keys regularly, and review each provider's data retention policies.</dd>

<dt>Local Provider (Ollama)</dt>
<dd>All processing is local. Keep Ollama updated and use disk encryption for sensitive data.</dd>

<dt>Enterprise Providers (Bedrock, Azure OpenAI)</dt>
<dd>Use IAM roles or managed identities instead of API keys, enable audit logging (CloudTrail or Azure Monitor), configure private endpoints, and use customer-managed encryption keys where available.</dd>
</dl>
