# AI Assistant Configuration

<Experimental />

Configure the Atmos AI Assistant to use different AI providers and customize their settings.

## Quick Start

Enable AI features with a single provider:

<File title="atmos.yaml">
```yaml
settings:
  ai:
    enabled: true
    default_provider: "anthropic"
    providers:
      anthropic:
        model: "claude-sonnet-4-5-20250929"
        api_key_env: "ANTHROPIC_API_KEY"
        max_tokens: 4096
```
</File>

<Terminal title="shell">
```bash
export ANTHROPIC_API_KEY="your-api-key-here"
atmos ai chat
```
</Terminal>

## Multi-Provider Configuration

Atmos supports 7 AI providers. Configure multiple and switch between them with **Ctrl+P** in the TUI,
or set `default_provider` for CLI commands.

<File title="atmos.yaml">
```yaml
settings:
  ai:
    enabled: true
    default_provider: "anthropic"

    providers:
      # Cloud Providers
      anthropic:
        model: "claude-sonnet-4-5-20250929"
        api_key_env: "ANTHROPIC_API_KEY"
        max_tokens: 4096

      openai:
        model: "gpt-5"
        api_key_env: "OPENAI_API_KEY"
        max_tokens: 4096

      gemini:
        model: "gemini-2.5-flash"
        api_key_env: "GEMINI_API_KEY"
        max_tokens: 8192

      grok:
        model: "grok-4"
        api_key_env: "XAI_API_KEY"
        max_tokens: 4096
        base_url: "https://api.x.ai/v1"

      # Local Provider (no API key needed)
      ollama:
        model: "llama3.3:70b"
        base_url: "http://localhost:11434/v1"

      # Enterprise Providers
      bedrock:
        model: "anthropic.claude-sonnet-4-5-20250929-v1:0"
        base_url: "us-east-1"              # AWS region
        # Uses AWS credentials (no api_key_env needed)

      azureopenai:
        model: "gpt-4o"                    # Your Azure deployment name
        api_key_env: "AZURE_OPENAI_API_KEY"
        base_url: "https://your-resource.openai.azure.com"
        api_version: "2024-02-15-preview"
```
</File>

:::info
For detailed setup instructions, hardware requirements, and pricing for each provider, see [AI Providers](/ai/providers).
:::

## Migration from Old Configuration

The old single-provider format (`provider:` + `model:` at the top level) still works but limits you
to one provider. Migrate to the `providers:` map for multi-provider support:

<File title="atmos.yaml">
```yaml
# Old format (still works)
settings:
  ai:
    enabled: true
    provider: "anthropic"
    model: "claude-sonnet-4-5-20250929"
    api_key_env: "ANTHROPIC_API_KEY"

# New format (recommended)
settings:
  ai:
    enabled: true
    default_provider: "anthropic"
    providers:
      anthropic:
        model: "claude-sonnet-4-5-20250929"
        api_key_env: "ANTHROPIC_API_KEY"
```
</File>

## Global Settings

<File title="atmos.yaml">
```yaml
settings:
  ai:
    enabled: true                         # Enable AI features (required)
    default_provider: "anthropic"         # Default provider for CLI commands
    send_context: false                   # Send stack configs to AI (default: false)
    prompt_on_send: true                  # Prompt before sending context (default: true)
    timeout_seconds: 60                   # Request timeout in seconds (default: 60)
    max_context_files: 10                 # Maximum stack files to send (default: 10)
    max_context_lines: 500                # Maximum lines per file (default: 500)
    max_history_messages: 20              # Max conversation messages (default: 0 = unlimited)
    max_history_tokens: 8000              # Max tokens in history (default: 0 = unlimited)
    default_skill: "general"              # Default AI skill fallback (default: "general")
```
</File>

<dl>
  <dt>`enabled`</dt>
  <dd>Enable or disable AI features (default: `false`)</dd>

  <dt>`default_provider`</dt>
  <dd>Default AI provider: `anthropic`, `openai`, `gemini`, `grok`, `ollama`, `bedrock`, or `azureopenai`</dd>

  <dt>`send_context`</dt>
  <dd>Automatically send stack configurations to AI for context-aware answers (default: `false`)</dd>

  <dt>`prompt_on_send`</dt>
  <dd>Prompt user before sending context, even when `send_context` is `true` (default: `true`)</dd>

  <dt>`max_history_messages`</dt>
  <dd>Maximum conversation messages to keep. Simple message-based limit (default: `0` = unlimited)</dd>

  <dt>`max_history_tokens`</dt>
  <dd>Maximum tokens in conversation history. More precise, approximate counting (~10-20% accuracy). If both limits are set, whichever is hit first wins (default: `0` = unlimited)</dd>
</dl>

### Provider-Specific Settings

Each provider in the `providers` map supports:

<dl>
  <dt>`model`</dt>
  <dd>Model to use (defaults vary by provider)</dd>

  <dt>`api_key_env`</dt>
  <dd>Environment variable containing the API key (not needed for Ollama or Bedrock with IAM)</dd>

  <dt>`max_tokens`</dt>
  <dd>Maximum tokens per response (defaults vary by provider). Roughly 1000 tokens = 750 words.</dd>

  <dt>`base_url`</dt>
  <dd>Custom API endpoint (required for Grok, Azure OpenAI; AWS region for Bedrock; optional for Ollama remote)</dd>

  <dt>`timeout_seconds`</dt>
  <dd>Request timeout in seconds (default: `60`, increase for large local models)</dd>
</dl>

## Session Management

Sessions store conversation history locally in SQLite so you can resume conversations across CLI invocations.

<File title="atmos.yaml">
```yaml
settings:
  ai:
    sessions:
      enabled: true                    # Enable persistent sessions (default: false)
      storage: "sqlite"                # Storage backend (default: sqlite)
      path: ".atmos/sessions"          # Storage location (default: .atmos/sessions)
      max_sessions: 10                 # Maximum sessions to keep (default: 10)
      auto_save: true                  # Auto-save messages (default: true)
      retention_days: 30               # Delete sessions older than N days (default: 30)

      # Auto-compact (optional) - summarizes old messages instead of dropping them
      auto_compact:
        enabled: true                  # Enable auto-compact (default: false)
        trigger_threshold: 0.75        # Trigger at 75% of max_history_messages
        compact_ratio: 0.4             # Compact 40% of messages
        preserve_recent: 10            # Always keep last 10 messages
        use_ai_summary: true           # Use AI for summaries (default: true)
        summary_model: "claude-3-haiku-20240307"  # Faster/cheaper model
```
</File>

<Terminal title="shell">
```bash
# Start a named session
atmos ai chat --session vpc-refactor

# Resume the same session later
atmos ai chat --session vpc-refactor

# List and clean sessions
atmos ai sessions
atmos ai sessions clean --older-than 30d
```
</Terminal>

Sessions are stored locally at `<project-root>/.atmos/sessions/sessions.db`. No session data is sent to
AI providers except during active conversations.

For detailed session management including auto-compact configuration, see [AI Sessions](/ai/sessions).

## Conversation History Management

By default, Atmos sends full conversation history with each request. For long conversations, this can cause
rate limiting and higher costs. Three approaches to manage it:

| Approach | Config | Best For |
|----------|--------|----------|
| **Simple truncation** | `max_history_messages: 20` | Most users, easy to configure |
| **Auto-compact** | `sessions.auto_compact.enabled: true` | Long conversations, preserves context |
| **Both** | Truncation + auto-compact | Maximum control, safety net |

Auto-compact summarizes old messages using AI instead of dropping them, preserving important context.
See [Auto-Compact](/ai/sessions#auto-compact-extended-conversations) for details.

:::warning
Setting history too low (4-6 messages) causes the AI to lose important context. Start with 20-30.
:::

## Tool Execution

Enable AI to inspect your infrastructure using read-only Atmos commands.

<File title="atmos.yaml">
```yaml
settings:
  ai:
    tools:
      enabled: true                        # Enable tool execution (default: false)
      require_confirmation: true           # Prompt before executing (default: true)
      allowed_tools:                       # Execute without confirmation
        - atmos_describe_*
        - atmos_list_*
        - atmos_validate_*
      restricted_tools:                    # Always require confirmation
        - file_edit
        - atmos_terraform_plan
      blocked_tools:                       # Never execute
        - atmos_terraform_destroy
        - atmos_terraform_apply
      yolo_mode: false                     # Skip all checks (DANGEROUS!)
```
</File>

For tool details, permission flow, and security best practices, see [AI Tools](/ai/tools).

## AI Skills

Install specialized skills from the marketplace or create custom ones. Skills provide focused expertise
for different tasks (Terraform, security, validation, etc.).

<Terminal title="shell">
```bash
# Install all 21+ official Atmos skills
atmos ai skill install cloudposse/atmos

# Switch skills with Ctrl+A in the TUI
```
</Terminal>

Custom skills are defined inline in `atmos.yaml`:

<File title="atmos.yaml">
```yaml
settings:
  ai:
    default_skill: "atmos-stacks"
    skills:
      cost-optimizer:
        display_name: "Cost Optimizer"
        description: "Analyzes infrastructure for cost savings"
        category: "analysis"
        system_prompt: |
          You are a cost optimization expert for Atmos infrastructure.
          Focus on overprovisioned resources, rightsizing, and unused resources.
        allowed_tools:
          - atmos_describe_component
          - atmos_list_stacks
          - read_stack_file
```
</File>

For complete skill documentation, see [AI Skills](/ai/skills) and [Skill Marketplace](/ai/skill-marketplace).

## LSP Integration

Atmos AI can validate YAML, Terraform, and HCL files using Language Server Protocol for precise error diagnostics.

<File title="atmos.yaml">
```yaml
settings:
  lsp:
    enabled: true
    servers:
      yaml-ls:
        command: "yaml-language-server"
        args: ["--stdio"]
        filetypes: ["yaml", "yml"]

  ai:
    use_lsp: true                      # Allow AI to use LSP (default: false)
    tools:
      enabled: true
```
</File>

For complete LSP documentation, see [LSP Integration](/lsp/lsp-client).

## Project Instructions

The AI uses project-specific context from `ATMOS.md` to provide consistent, context-aware responses.

<File title="atmos.yaml">
```yaml
settings:
  ai:
    instructions:
      enabled: true                        # Enable project instructions
      file: "ATMOS.md"                     # Instructions file location
      auto_update: false                   # Manual updates only (recommended)
      create_if_missing: true              # Auto-create template
```
</File>

For details on the instructions file structure and best practices, see [Project Instructions](/ai/instructions).

## Context-Aware Queries

When you ask about your specific infrastructure, Atmos can include stack configuration files in the AI request.
Control this with `send_context` and `prompt_on_send` in the global settings, or with environment variables:

<Terminal title="shell">
```bash
# Always send context without prompting
export ATMOS_AI_SEND_CONTEXT=true

# Never send context
export ATMOS_AI_SEND_CONTEXT=false
```
</Terminal>

By default (`send_context: false`), the AI prompts interactively when it detects your question needs
infrastructure context (keywords like "my stacks", "this repo", "describe the").

## Security Best Practices

- Never commit API keys to version control -- use environment variables
- Rotate keys per provider recommendations
- Set spending limits in provider dashboards
- Monitor usage to detect anomalies
