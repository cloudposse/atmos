# AI Skill Marketplace

<Experimental />

:::info Not Claude's Marketplace
The Atmos Skill Marketplace is **not related to Anthropic's Claude marketplace**. This is Atmos's own system for installing and sharing specialized AI skills from GitHub repositories. These skills work within the Atmos ecosystem to provide domain-specific expertise for infrastructure management. Skills follow the [Agent Skills open standard](https://agentskills.io) governed by the Linux Foundation's Agentic AI Foundation (AAIF).
:::

:::caution Coming Soon
The Skill Marketplace infrastructure is implemented and ready to use. However, community-contributed skills are still in development. You can create and share your own skills today, but the official catalog of community skills is coming soon.
:::

The Atmos AI Skill Marketplace enables you to install, share, and use community-created AI skills from GitHub repositories. Extend Atmos with specialized skills created by the community or publish your own custom skills for others to use.

## Overview

The Skill Marketplace provides:

- **Easy Installation** - Install skills directly from GitHub with a single command
- **Security Validation** - Automatic validation of skill metadata and version compatibility
- **Community Sharing** - Publish and discover skills created by the Atmos community
- **Local Management** - Track installed skills in a local registry at `~/.atmos/skills/`
- **Version Control** - Install specific versions using Git tags or branches
- **Safety Features** - Confirmation prompts for skills requesting destructive tool access

### How It Works

1. **Skill Repositories**: Developers create skills with `SKILL.md` files containing metadata and prompts
2. **GitHub Hosting**: Skills are published as public GitHub repositories
3. **Installation**: Users install skills with `atmos ai skill install <github-url>`
4. **Local Registry**: Installed skills are tracked in `~/.atmos/skills/registry.json`
5. **Automatic Loading**: Installed skills appear in the TUI skill selector (Ctrl+A)

## Installing Skills

### From GitHub

Install skills using various GitHub URL formats:

<Terminal title="terminal">
```bash
# GitHub shorthand (recommended)
atmos ai skill install github.com/cloudposse/atmos-skill-terraform

# With specific version tag
atmos ai skill install github.com/cloudposse/atmos-skill-terraform@v1.2.3

# With branch name
atmos ai skill install github.com/cloudposse/atmos-skill-terraform@main

# Full HTTPS URL
atmos ai skill install https://github.com/cloudposse/atmos-skill-terraform.git

# SSH URL
atmos ai skill install git@github.com:cloudposse/atmos-skill-terraform.git
```
</Terminal>

### Installation Process

When you install a skill, Atmos:

1. **Downloads** the repository from GitHub
2. **Validates** the SKILL.md file structure and metadata
3. **Checks** version compatibility with your Atmos version
4. **Prompts** for confirmation (shows tool access and permissions)
5. **Installs** to `~/.atmos/skills/github.com/<owner>/<repo>/`
6. **Registers** in the local skill registry

### Installation Options

#### Force Reinstall

Reinstall a skill that's already installed:

<Terminal title="terminal">
```bash
atmos ai skill install github.com/cloudposse/atmos-skill-terraform --force
```
</Terminal>

#### Skip Confirmation

Install without interactive confirmation prompt:

<Terminal title="terminal">
```bash
atmos ai skill install github.com/cloudposse/atmos-skill-terraform --skip-confirm
```
</Terminal>

:::warning Security Consideration
Only use `--skip-confirm` with skills from trusted sources. Always review skill metadata before installing, especially the `tools.allowed` section.
:::

# Skill: Terraform Expert

## Role

You are a specialized AI skill for Terraform component development,
debugging, and applying infrastructure-as-code best practices within
the Atmos framework.

## Your Expertise

- **Terraform Development** - Writing and maintaining Terraform modules
- **Component Architecture** - Designing reusable Atmos components
- **Best Practices** - Applying HashiCorp and Cloud Posse patterns
- **Debugging** - Troubleshooting plan/apply errors
- **Refactoring** - Improving existing Terraform code quality

## Instructions

### Analysis Workflow

When a user asks you to analyze or improve Terraform code:

1. **Read the component** - Use `read_component_file` to examine main.tf, variables.tf, outputs.tf
2. **Understand context** - Use `atmos_describe_component` to see how component is used in stacks
3. **Identify improvements** - Check for:
   - Outdated syntax
   - Missing required providers version constraints
   - Hardcoded values that should be variables
   - Missing outputs
   - Poor resource naming
   - Security issues

4. **Propose changes** - Explain each improvement before implementing
5. **Implement incrementally** - Make one change at a time
6. **Validate** - Run `terraform validate` after changes

### Coding Standards

Apply these Terraform best practices:

**Resource Naming:**
- Use underscores for resource names: `aws_instance.web_server`
- Descriptive names: `aws_s3_bucket.artifact_storage` not `aws_s3_bucket.bucket`

**Variables:**
- Always include description
- Set appropriate types (string, number, bool, list, map)
- Use sensible defaults where applicable
- Group related variables with blank lines

**Outputs:**
- Include description for all outputs
- Export all resources that other components might reference

**Modules:**
- Pin module versions: `version = "~> 5.0"`
- Use semantic versioning

### Tool Usage

- **Read before write**: Always read files before editing
- **Validate changes**: Run `terraform validate` after modifications
- **Test incrementally**: Make small changes and verify each one
- **Explain reasoning**: Tell the user why you're making each change

### Response Style

- Be concise but thorough
- Show code examples for recommendations
- Use Terraform terminology correctly
- Cite official Terraform documentation when relevant
- Format code blocks with ```hcl or ```terraform

**From Terraform Best Practices:**

- Use workspaces appropriately
- Apply the principle of least privilege for IAM

- Tag all resources consistently

## Validation Rules

Atmos validates all skills during installation. Your skill must pass these checks:

### Metadata Validation

#### Required Fields
- `name` must be present and kebab-case
- `display_name` must be present
- `version` must be valid semantic version
- `author` must be present
- `description` must be present
- `category` must be one of: `general`, `analysis`, `refactor`, `security`, `validation`, `optimization`
- `atmos.min_version` must be present and valid
- `repository` must be present

#### Tool Configuration
- Tool names must be valid Atmos tools
- No tool can appear in both `allowed` and `restricted` lists

### File Validation

#### SKILL.md
- File must exist with valid YAML frontmatter
- Frontmatter must be surrounded by `---` delimiters
- Markdown body must not be empty
- File should be under 100KB for optimal performance

### Version Compatibility

#### Atmos Version Checks
- Your Atmos version must be &gt;= `atmos.min_version`
- If `atmos.max_version` specified, your version must be &lt;= max version

**Example validation error:**

```
Error: skill validation failed: skill requires Atmos >= 1.50.0,
but current version is 1.48.0
```

**Fix:** Upgrade Atmos or use an older version of the skill:

<Terminal title="terminal">
```bash
# Upgrade Atmos
brew upgrade atmos

# Or install older skill version
atmos ai skill install github.com/user/skill@v0.9.0
```
</Terminal>

## Security Considerations

### Tool Access Control

Skills declare which tools they need access to. Review these carefully:

#### Safe Tools (Read-only)
```yaml
tools:
  allowed:
    - atmos_describe_component      # Read component config
    - atmos_list_stacks            # List stacks
    - read_file                    # Read files
    - read_component_file          # Read component files
    - search_files                 # Search in files
```

#### Potentially Dangerous Tools
```yaml
tools:
  allowed:
    - edit_file                    # Modifies files
    - write_file                   # Creates files
    - execute_bash                 # Executes commands
    - atmos_terraform_apply        # Applies Terraform
    - atmos_terraform_destroy      # Destroys infrastructure
```

:::danger Review Before Installing
**Always review skill source code on GitHub before installing skills that request:**
- File write access (`edit_file`, `write_file`)
- Command execution (`execute_bash`)
- Infrastructure changes (`terraform_apply`, `terraform_destroy`, `helmfile_apply`)
:::

### Restricted Tools

Use the `restricted` field to require confirmation for sensitive operations:

<File title="SKILL.md frontmatter">
```yaml
tools:
  allowed:
    - read_file
    - edit_file
    - execute_bash

  restricted:
    - edit_file        # User must confirm before editing
    - execute_bash     # User must confirm before executing
```
</File>

This creates a two-tier safety system:
1. **Allowed** - Skill can use these tools
2. **Restricted** - Skill can use, but user must confirm each usage

### Security Best Practices

**For Users:**
- Only install skills from trusted sources
- Review skill repository on GitHub before installing
- Check tool access in SKILL.md frontmatter
- Read skill prompt to understand behavior
- Start with `--skip-confirm=false` (default) until you trust a skill
- Be cautious with skills requesting `terraform_apply`, `execute_bash`, or `edit_file`

**For Skill Authors:**
- Request minimal tool access (principle of least privilege)
- Mark destructive tools as `restricted`
- Document all tool usage in prompt
- Include clear descriptions in frontmatter
- Add security warnings in README
- Keep prompts focused and transparent

## Marketplace Skill vs Built-in Skill vs Custom Skill

Atmos has three types of skills:

| Feature | Built-in Skills | Marketplace Skills | Custom Skills |
|---------|----------------|-------------------|---------------|
| **Location** | Compiled into Atmos binary | `~/.atmos/skills/` | `atmos.yaml` config |
| **Installation** | Pre-installed | `atmos ai skill install` | Add to config file |
| **Updates** | Atmos releases | Git pull/reinstall | Edit config |
| **Sharing** | Everyone has them | Share via GitHub | Team only (via config repo) |
| **Examples** | General, Stack Analyzer | Community contributions | Company-specific |
| **Format** | `pkg/ai/skills/builtin/*/SKILL.md` | Repository's `SKILL.md` | Inline in `atmos.yaml` |
| **Version Control** | Atmos version | Git tags/branches | Config file changes |

### When to Use Each

**Use Built-in Skills:**
- Standard Atmos workflows
- Proven, tested skills
- No installation needed
- Examples: Stack Analyzer, Security Auditor

**Use Marketplace Skills:**
- Specialized community tools
- Reusable across teams/organizations
- Version-controlled skills
- Examples: Cost optimizer, compliance checker, cloud provider specialists

**Use Custom Skills:**
- Company-specific workflows
- One-off experimental skills
- Team-internal tools
- Quick prototypes

## Troubleshooting

### Skill Not Found After Installation

**Symptom:** Skill installed successfully but doesn't appear in TUI (Ctrl+A)

**Cause:** Skill may be disabled or registry is corrupted

**Solution:**
<Terminal title="terminal">
```bash
# List skills to check status
atmos ai skill list

# Check registry file
cat ~/.atmos/skills/registry.json

# Restart Atmos AI session
atmos ai chat
```
</Terminal>

### Version Compatibility Error

**Symptom:**
```
Error: skill requires Atmos >= 1.50.0, but current version is 1.48.0
```

**Solution:**

<Terminal title="terminal">
```bash
# Option 1: Upgrade Atmos
brew upgrade atmos

# Option 2: Install older skill version
atmos ai skill install github.com/user/skill@v0.9.0

# Check compatible versions in skill's README
```
</Terminal>

### Invalid Metadata Error

**Symptom:**
```
Error: invalid skill metadata: display_name is required
```

**Cause:** Skill repository has malformed SKILL.md frontmatter

**Solution:**
- Report issue to skill author
- Check skill's GitHub issues for known problems
- Try different version: `@v1.0.0` instead of `@main`

### Tool Access Error

**Symptom:** Skill can't use certain tools

**Cause:** Tool not in skill's `allowed` list

**Solution:**
- This is intentional - skill author restricted access
- Fork the skill and modify SKILL.md if needed (use at your own risk)
- Request author to add tool access via GitHub issue

### Skill Installation Fails

**Symptom:** Download or installation errors

**Solution:**

<Terminal title="terminal">
```bash
# Check internet connection
curl -I https://github.com

# Verify GitHub URL is correct
# Check if repository is public
# Ensure you have git installed
which git

# Try with HTTPS instead of SSH
atmos ai skill install https://github.com/user/repo.git
```
</Terminal>

### Registry Corruption

**Symptom:** `Error: registry corrupted` or JSON parse errors

**Solution:**

<Terminal title="terminal">
```bash
# Backup current registry
cp ~/.atmos/skills/registry.json ~/.atmos/skills/registry.json.backup

# Reset registry (removes all skill registrations, but keeps skill files)
rm ~/.atmos/skills/registry.json

# Reinstall skills
atmos ai skill install github.com/user/skill1
atmos ai skill install github.com/user/skill2
```
</Terminal>
