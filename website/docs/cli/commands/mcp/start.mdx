## Description

The `atmos mcp start` command starts the Atmos MCP server, enabling AI clients to query your infrastructure using Atmos tools.

The server supports two transport protocols:
- **stdio** (default) — Standard input/output for desktop apps like Claude Desktop, VS Code, and Cursor
- **http** — HTTP with Server-Sent Events (SSE) for remote access, shared servers, and CI/CD

## Usage

```shell
atmos mcp start [flags]
```

### Flags

<dl>
  <dt>`--transport`</dt>
  <dd>Transport type: `stdio` (default) or `http`</dd>

  <dt>`--host`</dt>
  <dd>Host to bind HTTP server (HTTP transport only). Default: `localhost`</dd>

  <dt>`--port`</dt>
  <dd>Port to bind HTTP server (HTTP transport only). Default: `8080`</dd>
</dl>

### Examples

```shell
# Start MCP server with stdio transport (default, for desktop clients)
atmos mcp start

# Start MCP server with HTTP transport
atmos mcp start --transport http

# Start HTTP server on custom host and port
atmos mcp start --transport http --host 127.0.0.1 --port 3000
```

## Transport Modes

<dl>
  <dt>How it works</dt>
  <dd>**stdio (default):** Client spawns Atmos as a subprocess; communication over stdin/stdout. **HTTP:** Atmos listens on an HTTP port with SSE.</dd>

  <dt>Best for</dt>
  <dd>**stdio:** Desktop apps, local development. **HTTP:** Remote access, shared servers, CI/CD.</dd>

  <dt>Multiple clients</dt>
  <dd>**stdio:** No (1:1 with client). **HTTP:** Yes.</dd>

  <dt>Network exposure</dt>
  <dd>**stdio:** None. **HTTP:** Requires security measures.</dd>
</dl>

### HTTP Endpoints

When running with `--transport http`, the server exposes:

<dl>
  <dt>`GET /sse`</dt>
  <dd>Server-Sent Events stream for client communication.</dd>

  <dt>`POST /message`</dt>
  <dd>JSON-RPC 2.0 message endpoint.</dd>

  <dt>`GET /health`</dt>
  <dd>Returns `{"status":"healthy","clients":N}`. Use for monitoring and liveness probes.</dd>
</dl>

:::danger Security Warning
The MCP server has no built-in authentication or encryption.

- **stdio transport** (default) is safe for local use — it runs as a subprocess with no network exposure.
- **HTTP transport** exposes your infrastructure data to anyone who can reach the port. Always add firewall rules, VPN, or a reverse proxy with auth before exposing it on a network.
:::
