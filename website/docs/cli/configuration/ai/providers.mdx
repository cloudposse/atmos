<Intro>
The `ai.providers` section configures connections to AI model providers. Atmos supports **API providers**
(Anthropic, OpenAI, Gemini, Grok, Ollama, Bedrock, Azure OpenAI) and **CLI providers** (Claude Code, OpenAI Codex, Gemini CLI) that reuse your existing subscription.
</Intro>

<Experimental />

## Configuration

<File title="atmos.yaml">
```yaml
ai:
  default_provider: "anthropic"
  providers:
    anthropic:
      model: "claude-sonnet-4-6"
      api_key: !env "ANTHROPIC_API_KEY"
      max_tokens: 4096
    openai:
      model: "gpt-4o"
      api_key: !env "OPENAI_API_KEY"
      max_tokens: 4096
```
</File>

## Provider Settings

Each provider in the `providers` map supports the following settings:

<dl>
  <dt>`model`</dt>
  <dd>Model identifier to use for this provider. Defaults vary by provider (see below).</dd>

  <dt>`api_key`</dt>
  <dd>API key for the provider. Use the `!env` YAML function to read from an environment variable (e.g., `api_key: !env "ANTHROPIC_API_KEY"`). Not required for Ollama (local) or Bedrock (uses AWS credentials).</dd>

  <dt>`max_tokens`</dt>
  <dd>Maximum tokens per response (default: `4096`). OpenAI reasoning models (o3, o4-mini) use `max_completion_tokens` internally -- Atmos handles this automatically.</dd>

  <dt>`base_url`</dt>
  <dd>Custom API endpoint. Required for Grok (`https://api.x.ai/v1`) and Azure OpenAI (`https://<resource>.openai.azure.com`). For Bedrock, set to the AWS region (e.g., `us-east-1`). Ollama defaults to `http://localhost:11434/v1`.</dd>

  <dt>`api_version`</dt>
  <dd>API version string. **Required** for Azure OpenAI (e.g., `2025-04-01-preview`).</dd>
</dl>

  <dd>API Key / Azure AD authentication. Enterprise compliance, Azure integration.</dd>
</dl>

## CLI Providers

CLI providers invoke a locally installed AI tool as a subprocess. No API key needed — the CLI tool uses your existing subscription.

<dl>
  <dt>**Claude Code** — `claude-code`</dt>
  <dd>Claude Pro/Max subscription. Full MCP support. Best for interactive development.</dd>

  <dt>**OpenAI Codex** — `codex-cli`</dt>
  <dd>ChatGPT Plus/Pro subscription. Full MCP support. Open source (Apache 2.0).</dd>

  <dt>**Gemini CLI** — `gemini-cli`</dt>
  <dd>Google account (free tier: 1K req/day). MCP blocked for personal accounts.</dd>
</dl>

### CLI Provider Settings

In addition to `model`, CLI providers support:

<dl>
  <dt>`binary`</dt>
  <dd>Path to the CLI binary. Optional — defaults to `exec.LookPath` (finds it on PATH).</dd>

  <dt>`max_turns`</dt>
  <dd>Maximum agentic turns per invocation (Claude Code only).</dd>

  <dt>`max_budget_usd`</dt>
  <dd>Budget cap per invocation in USD (Claude Code only).</dd>

  <dt>`full_auto`</dt>
  <dd>Enable automatic approval for file writes (Codex CLI only). When MCP servers are configured, `--dangerously-bypass-approvals-and-sandbox` is used automatically.</dd>

  <dt>`allowed_tools`</dt>
  <dd>List of tools Claude Code is allowed to use (Claude Code only).</dd>
</dl>

### CLI Provider Examples

<File title="atmos.yaml">
```yaml
ai:
  default_provider: "claude-code"
  providers:
    claude-code:
      max_turns: 10
      # max_budget_usd: 1.00
      # allowed_tools: ["Read", "Glob", "Grep"]

    codex-cli:
      model: "gpt-5.4-mini"
      full_auto: true

    gemini-cli:
      model: "gemini-2.5-flash"
```
</File>

### MCP Pass-Through

When `mcp.servers` is configured, CLI providers automatically pass MCP servers to the CLI tool:

- **Claude Code:** Temp `.mcp.json` via `--mcp-config`
- **Codex CLI:** Written to `~/.codex/config.toml` (backup/restore after exit)
- **Gemini CLI:** Written to `.gemini/settings.json` in cwd (MCP blocked for personal accounts)

Auth-requiring servers are wrapped with `atmos auth exec -i <identity>`. Toolchain PATH and `ATMOS_*` env vars are injected automatically.

## Provider Comparison

<dl>
  <dt>Setup</dt>
  <dd>**Cloud:** Easy (API key). **Ollama:** Medium (install + download model). **Enterprise:** Complex (cloud config).</dd>

  <dt>Cost</dt>
  <dd>**Cloud:** Per-token pricing. **Ollama:** Free. **Enterprise:** Per-token + cloud costs.</dd>

  <dt>Data Privacy</dt>
  <dd>**Cloud:** Sent to provider. **Ollama:** 100% local. **Enterprise:** Stays in your cloud.</dd>

  <dt>Offline</dt>
  <dd>**Cloud:** No. **Ollama:** Yes. **Enterprise:** No.</dd>

  <dt>Rate Limits</dt>
  <dd>**Cloud:** Yes. **Ollama:** No. **Enterprise:** Yes (configurable).</dd>

  <dt>Compliance</dt>
  <dd>**Cloud:** Provider-dependent. **Ollama:** Your control. **Enterprise:** SOC2, HIPAA, ISO.</dd>

  <dt>Private Network</dt>
  <dd>**Cloud:** No. **Ollama:** Yes. **Enterprise:** Yes (VPC/VNet).</dd>
</dl>

:::tip Cost Optimization
For budget-conscious usage, try Gemini (cheapest cloud) or Ollama (free). For balanced cost and quality, Claude Sonnet or GPT-4o work well. Enterprise providers can leverage committed-use discounts.
:::

  <dd>Code on mid-range hardware.</dd>

#### Performance Tips

Hardware recommendations:
- **Apple Silicon (M1/M2/M3/M4):** Handles `llama4` well with unified memory
- **16GB RAM:** Use `llama3.1:8b` or `mistral:7b`
- **32GB+ RAM:** Use `mixtral:8x7b` or `llama3.3:70b`
- **64GB+ RAM:** Use `llama4`
- **GPU acceleration** (NVIDIA/AMD/Apple Silicon) provides 10-50x faster inference; Ollama detects and uses it automatically

```yaml
  ollama:
    model: "llama4"
    max_tokens: 4096       # Smaller = faster
```

#### Remote Deployment

To enable remote access on the server, set `OLLAMA_HOST=0.0.0.0:11434` before starting Ollama.

:::caution
Binding to `0.0.0.0` exposes the Ollama API to all network interfaces. Ensure the server is behind a firewall or reverse proxy with authentication in production environments.
:::

<dl>
  <dt>`OLLAMA_HOST`</dt>
  <dd>Server bind address (default: `127.0.0.1:11434`).</dd>

  <dt>`OLLAMA_ORIGINS`</dt>
  <dd>CORS allowed origins (default: `*` — allow all).</dd>

  <dt>`OLLAMA_MODELS`</dt>
  <dd>Model storage path (default: `~/.ollama/models`).</dd>
</dl>

#### AWS Bedrock

Authentication uses the standard AWS SDK credential chain (IAM roles, profiles, environment variables):

<Terminal title="shell">
```bash
# Option 1: Environment variables
export AWS_ACCESS_KEY_ID="AKIA..."
export AWS_SECRET_ACCESS_KEY="..."
```

:::warning Bedrock Endpoints
Use the `global.` prefix for dynamic routing across regions, or the `anthropic.` prefix for a specific region. Cross-region inference profiles use IDs like `us.anthropic.claude-sonnet-4-6-v1:0`.

#### Azure OpenAI

:::warning Azure-Specific Configuration
The `model` field must be your **deployment name** from Azure Portal, not the OpenAI model name. Both `base_url` (your resource endpoint) and `api_version` are required. See [Azure OpenAI Documentation](https://learn.microsoft.com/en-us/azure/ai-services/openai/) for available API versions.
:::

## Token Caching

Token caching (prompt caching) reduces costs by reusing frequently-sent content like system prompts.

<dl>
  <dt>**Anthropic** — 90% discount</dt>
  <dd>Manual markers. Configuration required (see below).</dd>

  <dt>**OpenAI** — 50% discount</dt>
  <dd>Automatic. No configuration required.</dd>

  <dt>**Gemini** — Free (included)</dt>
  <dd>Automatic. No configuration required.</dd>

  <dt>**Grok** — 75% discount</dt>
  <dd>Automatic. No configuration required.</dd>

  <dt>**Bedrock** — Up to 90% discount</dt>
  <dd>Automatic. No configuration required.</dd>

  <dt>**Azure OpenAI** — 50-100% discount</dt>
  <dd>Automatic. No configuration required.</dd>

  <dt>**Ollama** — N/A</dt>
  <dd>Local processing, caching not applicable.</dd>
</dl>

Most providers implement caching automatically. Only Anthropic requires explicit configuration:

<File title="atmos.yaml">
```yaml
ai:
  providers:
    anthropic:
      model: "claude-sonnet-4-6"
      api_key: !env "ANTHROPIC_API_KEY"
      cache:
        enabled: true              # Enable prompt caching (default)
        cache_system_prompt: true  # Cache skill system prompt
        cache_project_instructions: true # Cache ATMOS.md content
```
</File>

<dl>
  <dt>`cache.enabled`</dt>
  <dd>Enable or disable prompt caching (default: `true`). Cached tokens cost 90% less for Anthropic.</dd>

  <dt>`cache.cache_system_prompt`</dt>
  <dd>Cache the skill system prompt across messages (default: `true`).</dd>

  <dt>`cache.cache_project_instructions`</dt>
  <dd>Cache the ATMOS.md project instructions content (default: `true`).</dd>
</dl>

:::tip Maximize Cache Savings
For Anthropic, keep your ATMOS.md content stable during conversations and use named sessions for extended work on the same topic. For all other providers, caching works transparently with no action required.
:::

## Multi-Provider Configuration

Configure multiple providers and switch between them. Set `default_provider` for CLI commands, and press `Ctrl+P` during a chat session to switch mid-conversation.

### Provider Isolation

Each provider maintains its own completely isolated conversation history. When you switch providers, the new provider only sees messages from its own thread. When you switch back, it picks up right where you left off. UI notifications like "Switched to..." are never sent to any provider.

## Security Best Practices

<dl>
<dt>Cloud Providers (Anthropic, OpenAI, Google, xAI)</dt>
<dd>Use environment variables for API keys (never commit to git), rotate keys regularly, and review each provider's data retention policies.</dd>

<dt>Local Provider (Ollama)</dt>
<dd>All processing is local. Keep Ollama updated and use disk encryption for sensitive data.</dd>

<dt>Enterprise Providers (Bedrock, Azure OpenAI)</dt>
<dd>Use IAM roles or managed identities instead of API keys, enable audit logging (CloudTrail or Azure Monitor), configure private endpoints, and use customer-managed encryption keys where available.</dd>
</dl>
