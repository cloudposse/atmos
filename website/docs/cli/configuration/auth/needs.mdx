<Intro>
Declare additional identities a Terraform component requires beyond the default, so that all AWS profiles are authenticated before Terraform runs.
</Intro>

## The Problem

When Terraform components use multiple AWS provider aliases for multi-account patterns (e.g., hub-spoke networking or
cross-account peering), each provider assumes a different IAM role. In CI environments with OIDC authentication, only the
primary identity's profile is written to the shared credentials file. The additional provider aliases fail because their
AWS profiles don't exist.

## Configuration

Use `auth.needs` in your component's stack configuration to declare additional identities beyond the default:

<File title="stacks/catalog/vpc-peering.yaml">
```yaml
components:
  terraform:
    vpc-peering:
      auth:
        needs:
          - plat/prod       # Additional account
          - plat/staging    # Additional account
```
</File>

The default identity is always authenticated as the primary — you don't need to list it in `needs`. Only list the
*extra* identities your component requires.

All identities listed in `auth.needs` must be defined in your [identities](/cli/configuration/auth/identities) configuration
(either globally in `atmos.yaml` or via component-level overrides).

## Behavior

<dl>
  <dt>`needs`</dt>
  <dd>
    A list of additional identity names that the component requires beyond the default. Before Terraform runs, Atmos
    authenticates the default identity as the primary, then authenticates every identity in `needs` and writes all
    profiles to the shared AWS credentials file.

    - The **default identity** is always the primary, setting `AWS_PROFILE` and default credential environment variables.
      You do not need to list it in `needs`.
    - If **no default identity** is configured, the first entry in `needs` becomes the primary.
    - All entries in `needs` are authenticated and their profiles are written to the shared credentials file, making
      them available for Terraform provider aliases.
    - The `--identity` CLI flag takes **precedence** over both the default identity and `auth.needs`.
    - Failures for additional (non-primary) identities are **non-fatal** — Atmos logs a warning and continues.
  </dd>
</dl>

## Example

A hub-spoke networking component that peers VPCs across three AWS accounts:

<File title="stacks/catalog/transit-gateway.yaml">
```yaml
components:
  terraform:
    transit-gateway:
      auth:
        needs:
          - spoke/production  # Accepts peering from production VPCs
          - spoke/staging     # Accepts peering from staging VPCs
      vars:
        hub_account_id: "111111111111"
        spoke_accounts:
          production: "222222222222"
          staging: "333333333333"
```
</File>

Each identity's AWS profile is available for the corresponding Terraform provider alias:

<File title="components/terraform/transit-gateway/providers.tf">
```hcl
provider "aws" {
  # Uses the default identity (network/hub) — automatically authenticated as primary
}

provider "aws" {
  alias   = "production"
  profile = "spoke/production"
}

provider "aws" {
  alias   = "staging"
  profile = "spoke/staging"
}
```
</File>
