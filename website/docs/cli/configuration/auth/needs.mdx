<Intro>
Declare the identities a Terraform component requires so that all AWS profiles are authenticated before Terraform runs.
</Intro>

## The Problem

When Terraform components use multiple AWS provider aliases for multi-account patterns (e.g., hub-spoke networking or
cross-account peering), each provider assumes a different IAM role. In CI environments with OIDC authentication, only the
primary identity's profile is written to the shared credentials file. The additional provider aliases fail because their
AWS profiles don't exist.

## Configuration

Use `auth.needs` in your component's stack configuration to declare all required identities:

<File title="stacks/catalog/vpc-peering.yaml">
```yaml
components:
  terraform:
    vpc-peering:
      auth:
        needs:
          - core/network    # Primary identity (first entry)
          - plat/prod       # Additional account
          - plat/staging    # Additional account
```
</File>

All identities listed in `auth.needs` must be defined in your [identities](/cli/configuration/auth/identities) configuration
(either globally in `atmos.yaml` or via component-level overrides).

## Behavior

<dl>
  <dt>`needs`</dt>
  <dd>
    A list of identity names that the component requires. Before Terraform runs, Atmos authenticates every identity in
    the list and writes their profiles to the shared AWS credentials file.

    - The **first entry** becomes the primary identity, setting `AWS_PROFILE` and default credential environment variables.
    - All **remaining entries** are authenticated and their profiles are written to the shared credentials file, making
      them available for Terraform provider aliases.
    - The `--identity` CLI flag takes **precedence** over `auth.needs` when specified.
    - Failures for additional (non-primary) identities are **non-fatal** — Atmos logs a warning and continues.
  </dd>
</dl>

## Example

A hub-spoke networking component that peers VPCs across three AWS accounts:

<File title="stacks/catalog/transit-gateway.yaml">
```yaml
components:
  terraform:
    transit-gateway:
      auth:
        needs:
          - network/hub      # Primary: manages the transit gateway
          - spoke/production  # Accepts peering from production VPCs
          - spoke/staging     # Accepts peering from staging VPCs
      vars:
        hub_account_id: "111111111111"
        spoke_accounts:
          production: "222222222222"
          staging: "333333333333"
```
</File>

Each identity's AWS profile is available for the corresponding Terraform provider alias:

<File title="components/terraform/transit-gateway/providers.tf">
```hcl
provider "aws" {
  # Uses the primary identity (network/hub)
}

provider "aws" {
  alias   = "production"
  profile = "spoke/production"
}

provider "aws" {
  alias   = "staging"
  profile = "spoke/staging"
}
```
</File>
